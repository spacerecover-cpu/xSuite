### 1. Percentage document discount on invoices is computed as a fixed amount (discountType hardcoded null)

- **File:** `src/lib/invoiceService.ts:457`
- **Severity:** 🔴 critical · **Verdict:** CONFIRMED · **Category:** financial-math

**What's wrong:** createInvoice (invoiceService.ts:457) and updateInvoice (invoiceService.ts:688) both call computeDocumentTotals with `discountType: null`, while `discountAmount: invoice.discount_amount`. computeDocumentTotals (taxDocumentService.ts:320-322) only applies percentage math when `input.discountType === 'percentage'`; otherwise it treats discountAmount as an absolute currency amount. Meanwhile discount_type IS persisted to the row (pickInvoicePersistFields, invoiceService.ts:168; spread into insert at 481), and CaseDetail forwards discount_type='percentage' from the modal (CaseDetail.tsx:951 update / 971 create). InvoiceFormModal's own preview uses correct percentage semantics (InvoiceFormModal.tsx:489-491) and exposes a fixed/percentage toggle (940-954). quotesService passes the real discount type (quotesService.ts:430, 618), so invoices are the outlier.

**Failure scenario:** Invoice subtotal $1000, user toggles Discount to Percentage and enters 10 (=10%), tax 5%. Modal previews discount $100, net $900, tax $45, total $945. On save, createInvoice computes documentDiscount as fixed $10 (taxDocumentService.ts:322), net $990, tax $49.50, and stores total_amount=$1039.50 with discount_type='percentage', discount_amount=10 — $94.50 higher than the reviewed total, and a row whose stored total contradicts its own discount_type+discount_amount. Converting a percentage-discounted quote to an invoice therefore changes the total.

**Suggested fix:** Pass the real discount type in both computeDocumentTotals calls, matching quotesService: `discountType: invoice.discount_type ?? null` at invoiceService.ts:457 (createInvoice) and :688 (updateInvoice).

**Verifier note:** Independently reproduced: invoiceService.ts:457 and :688 both literally pass `discountType: null`; taxDocumentService.ts:320-322 branches on `input.discountType === 'percentage'`; discount_type is persisted (line 168) and forwarded from CaseDetail (971/951); quotesService.ts:430/618 confirm invoices are the asymmetric outlier; modal toggle at 940-954 makes percentage a reachable input. Money is observably wrong. Critical severity is appropriate.

### 2. Webhook skips signature verification whenever PAYPAL_MODE is not 'live' (default 'sandbox'), allowing forged events to mutate any tenant

- **File:** `supabase/functions/paypal-webhook/index.ts:146`
- **Severity:** 🔴 critical · **Verdict:** CONFIRMED · **Category:** auth-bypass

**What's wrong:** PAYPAL_MODE defaults to 'sandbox' (line 105). Signature verification only runs inside `if (paypalMode === 'live')` (line 116); the else branch (lines 146-148) merely logs 'skipping webhook signature verification' and processes the event unauthenticated. The handler then trusts `event.resource.custom_id` as tenantId (line 163) and drives state changes (active/cancelled/paused/past_due) on tenant_subscriptions and tenants keyed only by that attacker-supplied id. Any deployment where PAYPAL_MODE is unset or not exactly 'live' accepts arbitrary unsigned POST bodies against any tenant named in custom_id.

**Failure scenario:** With PAYPAL_MODE unset/sandbox, an attacker POSTs {"event_type":"BILLING.SUBSCRIPTION.CANCELLED","resource":{"custom_id":"<victim-tenant-uuid>"}} to the public webhook URL. No signature is checked; tenant_subscriptions.status and tenants.subscription_status for the victim tenant are set to 'cancelled', revoking paid access. A crafted ACTIVATED event grants free active status.

**Suggested fix:** Do not branch verification on mode. Require PAYPAL_WEBHOOK_ID and verify the signature in all modes (sandbox has its own verify-webhook-signature endpoint), and reject the request when verification is not configured.

**Verifier note:** CONFIRMED. Line 105 defaults paypalMode to 'sandbox'; the verify block is gated by `if (paypalMode === 'live')` (line 116); the else at 146-148 only console.logs and falls through to state-changing switch cases (175-311) that key updates on event.resource.custom_id (line 163). Real security exposure; the only caveat is it assumes the production deploy is not set to PAYPAL_MODE=live, but the insecure default posture is the defect. Line 146 anchors the skip branch.

### 3. Payment webhook inserts into non-existent billing_invoices.amount_cents column, so paid subscriptions are never recorded

- **File:** `supabase/functions/paypal-webhook/index.ts:278`
- **Severity:** 🔴 critical · **Verdict:** CONFIRMED · **Category:** schema-mismatch

**What's wrong:** In the PAYMENT.SALE.COMPLETED handler the insert into billing_invoices sets `amount_cents: Math.round(amount * 100)` (line 278), but billing_invoices has no amount_cents column. PostgREST rejects the whole INSERT with an unknown-column error. The insert result is discarded — lines 271-282 do `await supabase.from('billing_invoices').insert({...})` with no `{ error }` destructure — so the failure is completely silent. The subsequent tenant_subscriptions/tenants status='active' updates still run, so the tenant flips to active while no billing invoice is ever persisted.

**Failure scenario:** PayPal delivers PAYMENT.SALE.COMPLETED for a real subscription charge. The billing_invoices insert is rejected because amount_cents is not a column; the error is swallowed (never captured); billing_invoices gets no row. Every successful subscription payment goes financially unrecorded while the tenant is still marked active.

**Suggested fix:** Map the amount to the real numeric columns (e.g. subtotal/total, currency) instead of amount_cents, and destructure { error } from the insert to log/handle failures rather than discarding the result.

**Verifier note:** CONFIRMED. Grep of src/types/database.types.ts returns 0 matches for amount_cents; billing_invoices (lines 1249-1362) exposes amount_due/amount_paid/subtotal/total/tax_amount/currency but no amount_cents. The insert at lines 271-282 is awaited with no error destructure, so PostgREST's rejection is silent while the tenant_subscriptions/tenants 'active' updates at lines 284-292 still execute. Line 278 is exact.

### 4. Terminal-phase guard on custody transfers never matches — transfers stay enabled on delivered/cancelled/closed cases

- **File:** `src/components/cases/ChainOfCustodyTab.tsx:64`
- **Severity:** 🟠 high · **Verdict:** CONFIRMED · **Category:** state-machine

**What's wrong:** TERMINAL_CASE_STATUSES = {'delivered','cancelled'} (line 64) is compared against caseStatus.toLowerCase() in isTerminalStatus() (lines 67-70). caseStatus is caseData.status (CaseDetail.tsx:1433), which holds the human-readable status NAME — transition_case_status and set_case_status both write `status = v_to_status.name` (migrations 20260704190411:324, 20260704192500:66). The actual terminal names are 'Data Delivered', 'Cancelled — Customer Declined', 'Cancelled — Unrecoverable', 'Closed — Device Returned', 'Closed — Media Disposed'; lowercased, none equals 'delivered' or 'cancelled'. So isTerminalStatus() always returns false, caseIsTerminal (line 155) is always false, and transferEnabled = canPerformActions (line 156). The Transfer Custody button (line 397, disabled={!transferEnabled}, tooltip 'Case is in a terminal phase') is never actually disabled for a terminal case. initiateCustodyTransfer (chainOfCustodyService.ts:462) is a direct INSERT into chain_of_custody_transfers with no server-side terminal check, so the dead client guard is the only gate.

**Failure scenario:** As a technician, open a case whose status is 'Data Delivered' or 'Closed — Device Returned'. The Transfer Custody button is enabled; clicking it inserts a pending custody transfer (transfer_status='pending_acceptance') for devices that have already been released/returned/disposed, mutating the forensic custody state of a terminal case that the guard was written to protect.

**Suggested fix:** Gate on the case phase TYPE, not the free-text status name. Pass the phase (caseStatuses.find(s=>s.id===status_id)?.type, already computed at CaseDetail.tsx:716) into the tab and check ['delivered','cancelled','closed','no_solution'].includes(phase). Optionally add a server-side terminal check in initiateCustodyTransfer / an RPC.

**Verifier note:** CONFIRMED by reading all cited code. Migration 20260704190411 lines 46-56 define the exact terminal names; caseStatus prop = caseData.status (name) at CaseDetail.tsx:1433, while the phase type is a separate status_id lookup (line 716). isTerminalStatus therefore returns false for every real terminal case, and the transfer INSERT has no other terminal gate. Severity high retained: it is a fully non-functional forensic control gate on a legal-grade custody surface, though blast radius is bounded (creates a pending transfer requiring a second custodian to accept, not an irreversible state change).

### 87. Delivery challan statutory number allocated non-atomically (mint-before-write): gap on transient failure, duplicate under same-batch concurrency

- **File:** `src/lib/deliveryChallanService.ts:147`
- **Severity:** ⚪ low · **Verdict:** PLAUSIBLE · **Category:** idempotency-numbering

**What's wrong:** issueDeliveryChallan() reads case_job_history via getIssuedChallan() (line 140), mints a serial with get_next_number (line 147), then appends the record with log_case_history (line 165). The number is consumed before the record is written, so if log_case_history fails the minted number is lost and a retry mints a new one, leaving a gap in the challan series. The existence check and the mint are not transactional and there is no DB-side lock or unique constraint on (case_id, batch_id), so two truly-concurrent checkouts of the same batch could both mint and both append.

**Failure scenario:** DeviceCheckoutModal issues a challan; log_case_history fails transiently (challanErr path, DeviceCheckoutModal.tsx line 197, checkoutDone already true so retry re-runs only issuance). getIssuedChallan still returns null (no history row written), so a fresh number is minted — the previously consumed number is an unexplained gap. **Suggested fix:** Move issuance into a single SECURITY DEFINER RPC that, in one transaction, re-checks for an existing delivery_challan_issued row for the batch (advisory lock or unique constraint on case_id+batch_id), mints the number, and writes the history row atomically.

**Verifier note:** Mechanism verified from the code (get_next_number at line 147 precedes log_case_history at line 165; no lock/unique constraint). Downgraded from medium to low: a numbering gap is a weak, largely-cosmetic outcome (get_next_number sequences can gap anyway and GST tolerates unique-but-gapped series), and the concurrency-duplicate path requires two simultaneous checkouts of the SAME batch within a few-ms read-then-write window, which the single-modal isSubmitting guard and the UI flow (batch created only at checkout) make very unlikely. Real but low-severity robustness/compliance issue, not a wrong-result or corruption bug.

### 89. recordLoanRepayment never clamps remaining_amount and completion is by installment count, so rounded installments over-collect and drive remaining_amount negative on the final installment

### 91. Amount-in-words minor units do not carry when rounding reaches the full factor

- **File:** `src/lib/pdf/engine/amountInWords.ts:74`

## Methodology

This audit was produced by a deterministic multi-agent workflow:

1. **Fan-out (29 finders).** Independent agents each owned one slice — by domain (cases/lifecycle, chain-of-custody, financial math, invoices, quotes, payments/advances, banking, tax/VAT, inventory, stock, suppliers/PO, payroll, HR/leave, customers/companies, PDF, documents, localization, numbering, notifications, edge functions) and by cross-cutting dimension (React hooks, auth/permissions/tenancy, contexts/theming, data-migration, async/data-fetch sweep, query-key/cache correctness, the largest UI flows). Each was instructed to report only bugs it verified by reading the code, with a concrete failure scenario and exact `file:line`.
2. **Adversarial verification.** Every candidate was handed to a second, independent agent told to *reject* by default, re-open the cited file, and confirm/reject each claim. Rejections were dropped.
3. **Dedupe + rank.** Findings deduped by file+line and normalized title, then ranked by severity.

No fixes were applied — this is a findings report. Each entry carries enough context (file, line, mechanism, reproduction, fix) to action independently.
