|----|-------|--------|--------|

The governing fact across all four: the mutable case/financial surface is `PERMISSIVE` `USING(true)`/`WITH CHECK(true)`, reachable via the anon-key REST API. Any gate written only in React/TS is bypassable by one PostgREST call — gates that matter must live in the DB (column-grant REVOKE, `SECURITY DEFINER` RPC, `CHECK`, trigger, or `RLS WITH CHECK`).

**RLS-prerequisite call-out.** Steps 4 and 6 are partially bypassable by design until step 3 lands: (a) `case_qa_checklists.status` and `case_recovery_attempts.result` are unconstrained free-text with `INSERT WITH CHECK(true)` — an anon caller forges a `'passed'` row then satisfies the gate; (b) `clone_drives.status='delivered'` is a separate forgeable release surface the `cases`-only guard misses; (c) the `cases` guard is `BEFORE UPDATE`-only, so a direct anon INSERT seeds `status='Delivered'`. Step 3 (a targeted, gate-specific hardening — **not** the broad `USING(true)`→tenant rewrite, which is the separate `High` backlog #10) closes all three. `C2` sidesteps the surface entirely (column REVOKE is independent of the row policy); `C1` is a serialization bug whose isolation already survives the bypass — both ship first with no RLS dependency.

## C1 — Silent diagnostics data loss
**Stages:** 5, 6, 10.
**Root cause.** (1) `diagnosticsService.upsertDeviceDiagnostics` sends flat component columns the live `device_diagnostics` table lacks (real cols: `{id,tenant_id,device_id,diagnostic_type,tool_used,result jsonb,notes,performed_by,...}`), via `.insert([diagnosticsData as never])` (`diagnosticsService.ts:106`) — the cast defeats `tsc`, so it passes the 0-error gate yet PostgREST 400s. (2) `DeviceFormModal.tsx:471-474` swallows the error (`// Don't fail the entire operation`). Net: `device_diagnostics`/`case_diagnostics` = 0 rows across 13 devices; `case_devices.photos`/`physical_damage` never written. `reportPDFService.mapDiagnosticsRow` already reads `result.heads_status` etc. — the reader was migrated to jsonb; only the writer was left behind.
**Current behavior.** Device saves, modal reports success, every per-component finding silently dropped; report PDF diagnostics section always empty.
**urgent_bug (S, no migration).** Pack component fields into `result` jsonb with the exact keys `mapDiagnosticsRow` reads; remove `as never`; set `performed_by` from `auth.getUser()`; unpack on read; stop swallowing (warn-and-continue, loud toast, no false success); persist `case_devices.physical_damage` (defer `photos` — `text[]`, no upload pipeline). Add partial `UNIQUE INDEX idx_device_diagnostics_device_unique ON device_diagnostics(device_id) WHERE deleted_at IS NULL` (else read-then-write upsert yields duplicates and both `.maybeSingle()` callers throw PGRST116). Fix the broken `getDiagnosticsWithDevice` embed (`diagnosed_by_user:profiles` → `performed_by_user:profiles!device_diagnostics_performed_by_fkey`). **Enforcement:** data-integrity bug, not access — tenant isolation already DB-enforced (`RESTRICTIVE`+tenant-fill trigger); no security claim. Validity (`INSERT WITH CHECK=true`) left to lab_grade `CHECK`s.
**lab_grade (L, additive).** Typed nullable component columns + `smart_data`/`imaging_stats` jsonb + optional `*_status` `CHECK` vocab (survives REST bypass); keep `result`+`notes`; reader prefers typed, falls back to jsonb for pre-migration rows; `case_diagnostics` rollup (note: `case_diagnostics.result` is **TEXT**). Regen types.
**Recommended:** urgent now, lab_grade fast-follow. Highest-leverage line is removing `as never`.
**Must-not-break:** report-reader key parity; `case_devices` write path; 0-error `tsc`/schema-drift; edit-load round-trip.
**Risks:** non-atomic upsert (mitigated by UNIQUE index); key-drift (test against call-site keys `DeviceFormModal.tsx:448-470`, not the interface — `smart_data`/`imaging_stats`/`head_map` are never sent); un-swallow changes UX.
**Tests:** integration write→SELECT asserts row + `performed_by`; key-parity; round-trip; error-surfacing; report-renders-non-empty; `tsc=0`.
**Reviewer: needs_revision** — no rule violations. Folded in: UNIQUE index, embed fix, explicit `performed_by`, call-site key-parity test, do **not** claim either tier populates `{{diagnosis.success_probability}}` (no recoverability field exists), `case_diagnostics.result` is TEXT.

## C2 — Intake security & consent
**Stages:** 3 (gates 8).
**Root cause.** (1) `case_devices.password` plain `text`, written verbatim (`CreateCaseWizard.tsx:445`, `DeviceFormModal.tsx:364`, `caseService.ts:144`), re-read cleartext into the browser (`useCaseQueries.ts:163`) and PDF (`pdf/dataFetcher.ts:101` `select('*')`). It is a customer-supplied **decryption key** — must stay reversible; encrypt-at-rest, not hash. (2) No NDA (`ndas` 0 rows, zero frontend), no identity verification (`id_number`/`id_type` unused, no `id_verified` cols), no destructive-attempt authorization. **Enforcement root cause:** `case_devices.password` column SELECT/INSERT/UPDATE granted to `anon`+`authenticated`; `USING(true)` policies — one REST call dumps every password. `pgcrypto` installed; `vault.secrets`=0 (key must be seeded). 4 plaintext secrets at rest now.
**urgent_bug (L).** Seed Vault key; `ADD COLUMN password_encrypted bytea`; migrate the 4 rows via `pgp_sym_encrypt`, round-trip-verify byte-identical, then null plaintext (keep column per no-DROP); `REVOKE SELECT/INSERT/UPDATE ON case_devices(password),(password_encrypted) FROM anon,authenticated`; `set_device_password`/`reveal_device_password` `SECURITY DEFINER` RPCs that re-check tenant+role internally and write `log_chain_of_custody(action_category='access')` before returning, same transaction. Strip `password` from payloads **in the same commit as the revoke** (PostgREST emits the column regardless → permission-denies intake otherwise); remove from `duplicateCase`; remove from `useCaseQueries`/`dataFetcher` selects; regen types. **Enforcement:** DB-mandatory — ciphertext SELECT revoked (column privilege independent of row policy), plaintext only via the reveal RPC; append-only backstop intact (`prevent_audit_mutation` fires only on UPDATE/DELETE).
**lab_grade (XL).** `sign_nda` RPC wiring `ndas`; `customers_enhanced` `id_verified/_at/_by`; new `case_authorizations` table; `SECURITY DEFINER` recovery-entry guard (NDA+identity+authorization); intake `chain_of_custody` creation event. Customer-facing pieces blocked on the portal-identity JWT migration — staff-attested until then.
**Recommended:** urgent first (stops the breach), lab_grade follow-on.
**Must-not-break:** password reversible end-to-end; 4 rows migrate without loss; staff reveal keeps working; portal never sees it; isolation+backstop intact; `tsc=0`.
**Risks:** Vault-key loss = permanent data-decryption loss (decide escrow before migrating); revoke breaks the 3 inline writers unless password stripped same-PR; `duplicateCase` re-inserts it; legacy plaintext column lingers (CI guard against `select('*')`); already-leaked secrets in `pdf_generation_logs`/emailed reports need scrubbing; viewers/sales lose access (release note).
**Tests:** pgTAP zero column SELECT for anon/authenticated; reversibility on the 4 rows; in-tenant REST projection cannot read plaintext; reveal writes exactly one immutable `access` row; frontend no longer projects the secret.
**Reviewer: needs_revision** — no rule violations, bypassability correct/DB-layer. Folded in: strip password same-commit-as-revoke; remove from `duplicateCase`; RPCs derive caller+tenant internally and write custody before returning same-tx; provision+escrow Vault key and verify before nulling; audit `pdf_generation_logs`/reports for the 4 leaked secrets; CI guard against `select('*')`; viewers/sales release note.

## C3 — No payment/QA gating; checkout bypasses the state machine
**Stages:** 11, 13, 14, 15 (reaches 8, 16).
**Root cause.** (1) `transition_case_status` (`SECURITY DEFINER`) validates only edge-exists + `role∈allowed_roles`; never reads `requires[]`. `CaseStageBanner.tsx:217` literally says *"These are advisory."* Balance computed (`caseFinanceService.ts:70`, base currency) but gates nothing. (2) `DeviceCheckoutModal`→`log_case_checkout` writes history + raw `UPDATE cases SET status='Delivered'`. **Seed correction:** `log_case_checkout` does **not** set `app.bypass_status_guard`, so the `BEFORE UPDATE` guard already RAISEs 42501 for non-`Delivered` starts — checkout is already broken there; the real holes are guard-is-UPDATE-only (INSERT seeds any status — `CreateCaseWizard` inserts `status:'Received'` directly), no-op `Delivered→Delivered` passes, and advisory `requires[]`. **Proven live:** 3 `Delivered` cases, 0 recovery attempts, 0 QA checklists, 1 invoice/1 payment across 10 cases. **Edge correction:** live edges are `qa→recovery`=`qa_failed_reason`, `recovery→qa`=`recovery_completed`. `caseStateMachineService.ts` is at `src/lib/`.
**Current behavior.** Cases reach `Delivered`/closed with no QA, no recovery record, non-zero balance, via either path; anon REST bypasses any UI gate.
**urgent_bug (M, co-requisite-heavy).** Add a requires-evaluation block to `transition_case_status` (RAISE `23514`); set `cases.actual_completion=now()` inside the delivered move (never written in `src/` today, so a `delivery_recorded` check would otherwise be unsatisfiable); rewrite `log_case_checkout` to `PERFORM transition_case_status(...)` instead of raw UPDATE; extend the `cases` guard to enforce QA/recovery on entry into completed/delivered. **Hard co-requisite:** ship a minimal QA-pass + recovery-attempt capture UI same PR (evidence tables 0-row, no writer). **Enforcement:** checks in the `SECURITY DEFINER` fn **and** a `BEFORE INSERT OR UPDATE` trigger on `cases`; close the evidence-forgery hole with `CHECK`+role-gated INSERT (`is_staff_user`) on both evidence tables (step 3); unify on `app.bypass_status_guard`, never a second flag.
**lab_grade (XL).** `tenants.require_payment_before_release boolean DEFAULT false` + optional tolerance; `case_qa_checklists.reviewed_by`; `close_or_deliver_case` enforcing QA+recovery+delivery+policy-gated payment (base currency) with a server-evaluated `release.override` writing a financial CoC/audit row; per-device checkout; populate `resolution`/`actual_completion`.
**Recommended:** urgent now **with** minimal capture; lab_grade immediately after; defer payment-before-release (business policy).
**Must-not-break:** append-only backstop; existing edges/roles + bypass window; **base-currency** math; tenant isolation; CI gates. Reviewer ripples: `MarkAsDeliveredModal` (`useCaseMutations.ts:301`) is a third `Delivered` writer that will hard-fail and is coupled to the clone-retention countdown; routing `log_case_checkout` through the machine newly emits `case.phase_changed.customer` notifications.
**Risks:** enforcing on 0-row evidence tables bricks deliveries without the capture UI; legacy cases strand without a grandfather policy; over-broad trigger; override abuse; evidence forgery without step 3.
**Tests:** pgTAP edge RAISEs without evidence/succeeds with; anon-REST direct `Delivered` UPDATE blocked + forged-`passed`-row blocked; `log_case_checkout` produces exactly one transition; payment policy with base-currency multi-currency case; forensic re-query asserts no `Delivered` without evidence.
**Reviewer: needs_revision** — no rule violations; right layer but partially bypassable as first specified. Folded in: don't require pre-existing `actual_completion`/checkout row for `delivery_recorded` (set it in the transition); close evidence forgery (step 3); `cases` backstop `BEFORE INSERT OR UPDATE`; enumerate `MarkAsDeliveredModal` + retention coupling; account for new notification; correct edge mapping; unify on `app.bypass_status_guard`.

## C4 — No recovered-file manifest / customer delivery-approval gate
**Stages:** 12 (gates 13). Net-new feature.
**Root cause.** (1) No manifest entity anywhere (table/service/UI all absent; the only proof-of-recovery surface is read-only `case_reports`). (2) No gate: both delivery paths (`log_case_checkout` raw UPDATE; `markAsDeliveredMutation`→`transition_case_status`) have zero acceptance/QA/payment checks; the surface is `USING(true)`. `case_reports` stores approval/sent/version in a `content` JSONB blob (only `status`/`report_number`/`title`/`case_id`/`created_by` are real cols) — so a manifest reusing it would be un-enforceable. **Grounding corrections:** `cases.recovery_outcome` is **not** a column (lives in `case_job_history` details) — put forward-compat `manifest_id` on the acceptance table; `markAsDeliveredMutation` routes through `transition_case_status` (`useCaseMutations.ts:290-307`). Portal is `sessionStorage` fake-login (no JWT), so `is_portal_user()` (reads `request.jwt.claims->>'portal_token'`) is never satisfied — acceptance can't be DB-attributed/scoped to a customer. Confirmed helpers: `get_next_number(p_scope)`, `emit_notification_event`.
**urgent_bug (M).** New append-only tenant-scoped `delivery_acceptances` (`accepted_by_kind CHECK ('customer','staff_attested')`, `manifest_id uuid NULL`, REVOKE UPDATE/DELETE + immutability trigger — never RLS `WITH CHECK`); rewrite `log_case_checkout` to assert a passing QA checklist (else `23514`) + `balance_due=0` (base currency) or role-gated `p_waiver_reason`, insert acceptance, set `Delivered` via the unified `app.bypass_status_guard` window; add a `BEFORE UPDATE` trigger on `clone_drives` rejecting `status='delivered'` outside the RPC. **Enforcement:** DB-enforced; caveat — acceptance is staff-attested (cannot be customer-scoped until the JWT migration) and the QA precondition is un-forgeable only with step 3.
**lab_grade (XL, multi-PR).** Typed `recovery_manifests`+`recovery_manifest_items` (finalize trigger locks items at `status='final'`); `recovery_manifest_acceptances` (`RESTRICTIVE` tenant isolation + an additional `RESTRICTIVE` portal-scope policy: `is_portal_user()` ⇒ `customer_id=JWT claim`; append-only); RPCs `build_recovery_manifest_from_clone`/`finalize_recovery_manifest`/`accept_recovery_manifest`/`release_case`; staff Manifest tab + `pdfmake` `RecoveryManifestDocument`; portal Accept/Dispute. **Dependency:** portal-identity JWT migration (its own PR) that also **defines tenant resolution for portal principals** before `is_portal_user()` can return true (else every `RESTRICTIVE tenant_isolation` policy mis-scopes app-wide). **Enforcement:** finalized manifests locked; acceptance portal-row-scoped; release gate RAISEs without accepted/waived; typed tables, never `case_reports.content`.
**Recommended:** urgent first (standalone release gate); lab_grade as a sequenced epic (JWT → manifests → acceptances → swap into `C3`'s RPC).
**Must-not-break:** existing machine + bypass pattern; clone-retention; append-only backstop pattern on new tables; reports stay functional; tenant isolation on every new table; `tsc=0`/schema-drift.
**Risks:** portal-identity change is security-sensitive and app-wide; `clone_drives` separate forgeable surface; evidence forgery until step 3; QA gate depends on 0-row `case_qa_checklists`; finalize/lock rigidity needs a versioning story; ingestion source underspecified.
**Tests:** pgTAP `release_case` RAISEs/succeeds; anon-REST forged-acceptance/finalized-item-edit/acceptance-mutation all fail; portal-scope (A sees only A); manifest lifecycle build/finalize/accept; checkout blocked until accepted/waived; regression + types regen; PDF snapshot incl. RTL.
**Reviewer: needs_revision** — no rule violations; right layer, not fully bypass-proof as first written. Folded in: `clone_drives` trigger; unify with `app.bypass_status_guard` and actually set it; `CHECK`+gated QA writes; `manifest_id` on `delivery_acceptances` (not `cases`); sequence JWT migration with portal tenant resolution first; typed tables + REVOKE/trigger immutability; full tenant-table requirements per new table.
