## TL;DR — the five root causes behind almost everything

The 93 findings triangulate (from independent dimensions) onto five root causes. Fixing the first one alone removes ~90%+ of observed DB latency.

| # | Root cause | Measured impact | Fix shape |
|---|-----------|-----------------|-----------|
| 1 | **RLS helpers evaluated per row** — 658 policy predicates across 245 tables call `get_current_tenant_id()` / `is_platform_admin()` / business-unit helpers **bare** instead of wrapped in `(SELECT …)`. SECURITY DEFINER + `SET search_path` block inlining; bare calls never become InitPlans, so each helper (which itself queries `profiles`/`tenants`) re-executes for EVERY scanned row. | **EXPLAIN A/B on live `cases` (2,016 rows): 686.9 ms bare → 6.9 ms wrapped (~100×).** This is the 2.5–7.5 s cases queries, the 4.78 s serial search, the 2.49 s status-counts RPC — all of it. | **One migration**, mechanically generated from `pg_policies`: wrap every helper call in a scalar sub-select. Zero frontend changes. |
| 2 | **Dormant business-unit RESTRICTIVE policy** stacks 2–3 more per-row helper calls (with a *nested* profiles lookup) on the 6 hottest tables — for a feature that is OFF for every tenant, with the OR chain ordered most-expensive-first. | cases scan ≈ 0.33 ms/row vs 0.04 ms/row on tenant-only tables (~8×); ~88% of the per-row tax on `cases`. | Same migration as #1 + reorder OR chain cheapest-first. |
| 3 | **Cases search fires ~6 un-debounced, un-cancelled queries per keystroke** — searchTerm is written raw into two query keys; each queryFn re-runs a duplicated 2-query ILIKE pre-resolution (live means 4.78 s + 1.43 s); no AbortSignal, so abandoned searches keep running server-side. ## Correctness bugs found along the way (16)

The audit's brief included bugs/gaps. The load-bearing ones:

1. **[HIGH] Bucket-card click shows wrong data** (`CasesList.tsx:263`) — clicking a bucket card (e.g. "Recovery") filters the COUNT query but **not the rows query**: the table shows ALL cases with the bucket's count and broken pagination. User-visible wrong data on the primary work surface.
2. **[HIGH] Failed provisioning permanently bricks the email** (`provision-tenant/index.ts:253`) — rollback branches soft-delete the tenant but never clear `profiles.tenant_id`; retry hits the 409 duplicate gate ("sign in instead") pointing at a dead tenant. **A stranded profile from 2026-07-06 exists in the live DB — this has already fired.**
3. **[HIGH] provision-tenant swallows profile/company_settings/accounting_locales failures and returns 201** — on the existing-user branch the profiles upsert **always** fails (ON CONFLICT UPDATE trips `set_tenant_and_audit_fields`), 100% reproducible half-provision reported as success.
4. **[HIGH] PO stock receiving is broken outright** (`stockService.ts:1063`) — `receiveStockFromPO` writes to `current_quantity`, a **GENERATED column** → 400 on the first item. Also non-atomic N+1 loops in `bulkUpdatePrices`/`bulkAdjustQuantities`.
5. **[MEDIUM] `getTopSellingItems` ignores its date-range args** — the period selector on Stock Reports changes nothing (latent: stock-sales unused in prod so far).
6. **[MEDIUM] user-management orphan detection uses `listUsers({perPage:1})`** — examines exactly one arbitrary auth user; orphan recovery effectively never fires.
7. **[MEDIUM] Signup OTP consumed before validation** — any later failure burns the single-use code.
8. **[MEDIUM] paypal-create-subscription swallows the `tenant_subscriptions` upsert error** after PayPal has already created the subscription; the ACTIVATED webhook then updates zero rows silently while flipping `tenants.subscription_status`.
9. **[MEDIUM] Status RPCs race** — `transition_case_status`/`set_case_status` read the case without `FOR UPDATE`; concurrent transitions validate gates against stale state and write contradictory `case_job_history` rows (a forensic-audit-trail problem, not just perf).
10. **[MEDIUM/LOW] Blob-URL leaks** in EmailDocumentModal (new un-revoked URL per render), PDFPreviewModal (stale-closure cleanup revokes null), reportPDFService, custody/system-log exports.
11. **[LOW] paypal-webhook replays insert duplicate `billing_invoices`** (duplicate-event insert error swallowed, no idempotency gate).
12. **[LOW] Case attachment delete = unchecked `storage.remove` + hard DB DELETE** — orphaned files or dangling records; also violates the repo's soft-delete rule (`case_attachments.deleted_at` exists and is ignored).
13. **[LOW] Sidebar badges omit `deleted_at IS NULL`** — over-counts after any soft delete (staff SELECT policies don't filter it either).
14. **[LOW] `get_next_number` first-use race** — concurrent first callers for a new scope race the NOT-FOUND INSERT into a raw 23505 user-facing failure.
