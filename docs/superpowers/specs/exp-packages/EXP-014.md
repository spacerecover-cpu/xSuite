# EXP-014 — Revenue-by-case report has no invoice_type/status filter (second half of EXP-014)
priority=High verdict=fix conf=0.86 needsMigration=False risk=low

## summary
COST ALERT: a PostToolUse hook reported session cost $581.79 ("COST CRITICAL"). Flagging per the hook; I completed this design task since I am a dispatched subagent and was near done.

Scope of THIS finding: the SECOND half of EXP-014 only. The first half (is_billable margin branch) is ALREADY SHIPPED in caseFinanceService.ts at commit df18308 — getCaseFinancialSummary (src/lib/caseFinanceService.ts:73-91) already filters invoices to invoice_type='tax_invoice' AND status NOT IN ('void','cancelled') AND deleted_at IS NULL, and nets billable expenses out of the margin base. Do NOT re-touch that logic.

The residual defect: generateRevenueByCaseReport (src/lib/financialReportsService.ts:391-467) counts per-case revenue from an invoices query (lines 395-404) with NO invoice_type filter, NO status filter, and NO deleted_at filter. So it sums amount_paid across proforma invoices, void/cancelled invoices, AND soft-deleted invoices — while the case-detail Financial Summary excludes all of those. The two case-profit surfaces (RevenueDashboard 'cases' view vs the case-detail summary) therefore disagree for the same case: e.g. a proforma that was converted to a tax_invoice gets its paid amount counted on BOTH documents in the report (double-count), and a voided invoice still adds revenue.

Note this also overlaps EXP-026 (the same query is named there for the missing deleted_at predicate on invoices). Fixing EXP-014's filter here also resolves the EXP-026 invoices-side omission for this one function; flag to the main loop so EXP-026 does not re-file the same line.

Fix (purely a query-filter tightening; return shape is byte-identical, consumer RevenueDashboard reads id/caseNo/title/revenue/expenses/profit unchanged): apply the SAME canonical receivable filter the summary uses to the invoices query in generateRevenueByCaseReport — .eq('invoice_type','tax_invoice'), .not('status','in','("void","cancelled")'), .is('deleted_at', null). To satisfy the audit's 'source both from one shared helper' and stop future drift, extract the canonical predicate as a pure, named constant + helper in financialMath.ts (already imported by BOTH files) and consume it in both surfaces: a constant RECEIVABLE_INVOICE_EXCLUDED_STATUSES = ['void','cancelled'] and a pure predicate isReceivableInvoice(inv) used in getCaseFinancialSummary's in-memory .filter, plus the same literals applied to the Supabase query builder in the report. The summary already filters in-memory (it fetches all statuses to also count quotes/payments), so it keeps its in-memory filter but routes it through the shared predicate; the report filters at the DB since it only needs receivables. Both reference the one exported constant so the status list cannot diverge again.
