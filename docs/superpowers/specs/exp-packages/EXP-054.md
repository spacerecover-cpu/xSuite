# EXP-054 — financial_transactions ledger broadly visible/insertable (SELECT USING(true), INSERT is_staff_user())
priority=High verdict=fix conf=0.88 needsMigration=True risk=high

## summary
EXP-054 is a within-tenant least-privilege defect on the append-only ledger. Live policies (verified): `financial_transactions_tenant_isolation` RESTRICTIVE (correct, keep), `financial_transactions_select` PERMISSIVE `USING(true)`, `financial_transactions_insert` PERMISSIVE `WITH CHECK (is_staff_user())`. Tenant isolation is intact; the gap is (a) any role incl. `viewer` reads every posted ledger line tenant-wide, and (b) any staff (technician/sales/hr) can directly insert a ledger row.

Verified blast radius before tightening INSERT: every money-bearing write goes through a SECURITY DEFINER RPC (`post_manual_transaction`, `reverse_financial_transaction`, `record_payment`, `record_expense_disbursement`, `create_receipt_with_allocations`, `convert_proforma_to_tax_invoice` — all `prosecdef=true` confirmed). Under SECURITY DEFINER, RLS INSERT WITH CHECK is bypassed, so tightening `_insert` does NOT touch those flows. The ONLY app paths that perform a *direct* (non-definer) INSERT — and are therefore subject to the policy — are: (1) `createFinancialTransaction` in src/lib/financialService.ts, whose sole caller is `approveExpense` (src/lib/expensesService.ts:426); approval is already gated to accounts-or-higher in the UI, and `has_role('accounts')` resolves to exactly owner/admin/manager/accounts (verified from the function body) — so the new policy matches the existing authorization with zero behavior change for legitimate approvers; (2) the generic CSV importer for the `revenue`/`transactions` entity configs (src/lib/importExportService.ts:176-203) which inserts into `financial_transactions` directly — importing is an admin/accounts action, so accounts-gating is appropriate and consistent.

DESIGN: migration-only change, no TS code change required. (1) Drop+recreate `financial_transactions_insert` as `WITH CHECK (has_role('accounts'))`. (2) Drop+recreate `financial_transactions_select` as `USING (is_staff_user())` so viewers lose ledger read while all working staff keep it. Keep the RESTRICTIVE tenant isolation untouched (it ANDs with these). This is the audit's recommended minimum and is the safe full scope for EXP-054. Idempotency unique index (EXP-001/EXP-022) and ledger taxonomy/category (EXP-029) are explicitly OUT of scope here — separate items — and are noted as dependencies only for ordering.

## migrationSql
```sql
-- EXP-054: tighten financial_transactions least-privilege (tenant isolation unchanged)
-- INSERT: only owner/admin/manager/accounts (has_role('accounts')) may directly insert a
-- ledger row. All money-bearing flows use SECURITY DEFINER RPCs and are unaffected; the only
-- direct-insert app paths are expense approval (already accounts-gated) and CSV import of
-- revenue/transactions (an accounts/admin action).
DROP POLICY IF EXISTS financial_transactions_insert ON public.financial_transactions;
CREATE POLICY financial_transactions_insert ON public.financial_transactions
  AS PERMISSIVE FOR INSERT TO authenticated
  WITH CHECK (has_role('accounts'));

-- SELECT: drop the open USING(true). Restrict to working staff (non-viewer) so read-only
-- viewers cannot read confidential ledger lines. ANDed with the RESTRICTIVE tenant isolation
-- policy, so cross-tenant reads remain impossible.
DROP POLICY IF EXISTS financial_transactions_select ON public.financial_transactions;
CREATE POLICY financial_transactions_select ON public.financial_transactions
  AS PERMISSIVE FOR SELECT TO authenticated
  USING (is_staff_user());

-- financial_transactions_tenant_isolation (RESTRICTIVE FOR ALL) is intentionally left intact.
```

## blastRadius
Could-break analysis, each mitigated: (1) Expense approval — `approveExpense`->`createFinancialTransaction` is the only non-definer ledger INSERT in app code; it is invoked only by accounts-or-higher (UI gate isAccountsRole) and `has_role('accounts')`=owner/admin/manager/accounts, so legitimate approvals are unaffected; a non-accounts user who somehow reached it now fails at the DB (correct hardening). (2) CSV import of revenue/transactions (importExportService entity configs at :176-203) does a direct insert — importing is an accounts/admin operation, so accounts-gating is consistent; a non-accounts importer would now be blocked at the ledger (acceptable, matches intent). (3) Invoices/payments/receipts/banking/disbursement — ALL go through SECURITY DEFINER RPCs (verified prosecdef=true for post_manual_transaction, reverse_financial_transaction, record_payment, record_expense_disbursement, create_receipt_with_allocations, convert_proforma_to_tax_invoice); RLS INSERT WITH CHECK is bypassed under definer, so these are untouched. (4) Reports/reads — financialReportsService, transactionsService.getTransactionStats, TransactionsList, RevenueDashboard all run as working staff (non-viewer accountants/managers) and keep SELECT; only `viewer` loses ledger read, which is the intended confidentiality fix. (5) Tenant isolation — the RESTRICTIVE policy is left intact and ANDs with the new permissive policies, so no cross-tenant exposure is introduced. (6) Platform admin — both is_staff_user() and the isolation policy already special-case is_platform_admin via the RESTRICTIVE clause; platform admins are unaffected. VAT/invoices have no direct dependency on these policies. Reversibility: pure policy swap, instantly revertible by recreating USING(true)/is_staff_user() if an unforeseen non-definer staff insert path surfaces.

### migrationSafety
Safe and additive-equivalent. The migration only swaps two PERMISSIVE policies (DROP POLICY IF EXISTS + CREATE) and explicitly leaves the RESTRICTIVE financial_transactions_tenant_isolation intact, so cross-tenant isolation is preserved (verified live: RESTRICTIVE FOR ALL with tenant_id = get_current_tenant_id() OR is_platform_admin()). No data rows touched, no column/type change, no DROP TABLE/COLUMN, no DELETE. Lock is a brief ACCESS EXCLUSIVE on pg_policy for the relation only — negligible on a 1-row table. Fully reversible by recreating USING(true)/is_staff_user(). New policies tighten (narrower than current), so they cannot grant access that did not exist. No database.types.ts regen needed. One residual: there is a tiny window where, between DROP and CREATE inside the same migration txn, the policy is absent — but since DDL runs in a transaction the change is atomic, so no exposure. The only real-world consequence is the intended one: viewers lose SELECT and non-accounts staff lose direct INSERT (incl. the CSV import path), which is a behavior change to flag, not a safety defect.
