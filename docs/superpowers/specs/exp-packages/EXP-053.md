# EXP-053 — expenses_select is USING(true): every staff role incl. viewer sees all expenses tenant-wide
priority=High verdict=fix conf=0.86 needsMigration=True risk=high

## summary
The PERMISSIVE `expenses_select` policy on `public.expenses` is `USING(true)` (verified live), ANDed only with the RESTRICTIVE `expenses_tenant_isolation`. Result: any authenticated tenant principal — including read-only `viewer` and non-financial staff — can read every staff member's expense rows (amounts, vendors, descriptions, notes, case links). The only role gate today is on the UI action buttons (`isAccountsRole` in ExpensesList.tsx:126), not on visibility.

SAFE SLICE (this design): replace `expenses_select` `USING(true)` with `is_staff_user() OR is_platform_admin()`. `is_staff_user()` (verified def) returns true for owner/admin/manager/technician/sales/accounts/hr and FALSE for `viewer`. This exactly closes the reported defect (read-only viewer reading financial data) while preserving full read access for every role that the existing unscoped aggregation readers depend on. `is_platform_admin()` is retained explicitly because `is_staff_user()` only checks tenant `profiles.role` and would otherwise drop platform admins (tenant_id IS NULL) — though the RESTRICTIVE tenant-isolation policy already grants them; keeping it makes the SELECT self-consistent with the other expense policies' intent and harmless (both must pass: RESTRICTIVE AND PERMISSIVE).

WHY NOT submitter-scoping (the audit's 'ideal'): I traced every `from('expenses')` SELECT outside expensesService — financialReportsService.ts (P&L, 6 sites), caseFinanceService.ts (case profitability, 4 sites), ReportsDashboard.tsx (2 sites), featureGateService.ts (usage count), importExportService.ts (count). ALL run UNSCOPED tenant-wide aggregations with no `created_by` predicate and rely on seeing every row. A `created_by = auth.uid() OR has_role('manager'|'accounts')` RLS SELECT would silently undercount expenses in P&L, case margins, and reports for any technician/sales/hr user (who pass none of those role checks but legitimately view reports) — a correctness regression worse than the confidentiality leak it fixes. Submitter-scoping is therefore DEFERRED to a coordinated effort that first moves those aggregations to SECURITY DEFINER RPCs (overlaps EXP-056 stats RPC work) so the read path no longer depends on broad row visibility.

This is migration-only on the DB side; no app code must change for correctness. Optional defensive UI: hide the Expenses sidebar/route for `viewer` so they get a clean denial instead of an empty/erroring grid — but that is cosmetic and gated separately.

## migrationSql
```sql
-- EXP-053: expenses_select was USING(true) — every staff role incl. read-only
-- viewer could read tenant-wide expense financials. Gate SELECT behind staff
-- membership. is_staff_user() excludes 'viewer'; the RESTRICTIVE
-- expenses_tenant_isolation policy still ANDs tenant scoping on top of this.
-- is_platform_admin() retained for self-consistency with sibling policies
-- (harmless: PERMISSIVE OR RESTRICTIVE-AND semantics preserve isolation).

DROP POLICY IF EXISTS expenses_select ON public.expenses;

CREATE POLICY expenses_select
  ON public.expenses
  AS PERMISSIVE
  FOR SELECT
  TO authenticated
  USING (is_staff_user() OR is_platform_admin());
```

## blastRadius
What could break: any code path that SELECTs from `expenses` while authenticated as a NON-staff principal (only `viewer` today, since portal users hit different surfaces). After this change a viewer's expense queries return [] instead of rows. Verified readers and their exposure: ExpensesList grid+stats (viewer sees empty state — acceptable, that is the fix), financialReportsService P&L (financialReportsService.ts:98 and 5 more sites), caseFinanceService case profitability (caseFinanceService.ts:53 and 3 more), ReportsDashboard (lines 236,294), featureGateService usage count (line 267), importExportService count (line 561) — EVERY ONE of these runs as whatever staff role opened the page; none are viewer-only flows, and all staff roles remain inside is_staff_user(), so their aggregations return the identical row set as before. That is precisely why I rejected created_by-scoping: it WOULD have silently undercounted these aggregations for technician/sales/hr. Tenant isolation: untouched — the RESTRICTIVE expenses_tenant_isolation policy still ANDs `tenant_id = get_current_tenant_id() OR is_platform_admin()` on top, so cross-tenant reads remain impossible regardless of staff status. Invoices/VAT/banking: not touched — `expenses_select` governs only the expenses table; vat_records, financial_transactions, invoices, payments have their own policies (financial_transactions' own USING(true) is EXP-054, a separate finding). Writes (insert/update/delete) are unaffected — those policies already gate on has_role('accounts'/'admin'). The design avoids regression by choosing the one role-set (is_staff_user) that is the exact superset of every existing expense reader minus the one role the finding says must lose access (viewer), so no aggregation loses a row.
