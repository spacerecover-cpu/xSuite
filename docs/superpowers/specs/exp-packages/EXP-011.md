# EXP-011 — updateExpense/transitions editable on any expense by accounts-or-higher; RLS lacks ownership/status predicate
priority=High verdict=fix conf=0.8 needsMigration=True risk=high

## summary
EXP-011 has two faces (audit 3.3 edit/transition authority; 3.7 ownership/visibility). LIVE DB confirms: expenses_update USING and CHECK both equal (SELECT has_role accounts) with NO status and NO created_by predicate; expenses_select equals USING(true) so every role including viewer reads all rows; expenses_delete equals has_role admin but governs only hard DELETE which the app never issues (dead path; soft-delete rides the UPDATE policy). RESTRICTIVE expenses_tenant_isolation is intact, so this is intra-tenant only. profiles.id IS the auth user id (no user_id column; handle_new_user sets profiles.id to auth.uid), and the frontend stamps created_by from profile.id, so created_by equals auth.uid is a valid RLS ownership predicate. Service updateExpense already blocks approved/paid/voided edits (EXP-006 shipped) but has no ownership check; approveExpense enforces pending-only plus segregation of duties (approver not equal creator, EXP-008); submit/reject/markPaid carry status guards. Fix is defense-in-depth: (1) MIGRATION tightens SELECT to is_staff_user plus submitter-or-approver scope and splits UPDATE into an approver policy keeping owner/admin/manager/accounts transition power plus a submitter policy (created_by equals auth.uid AND status in draft/pending/rejected). (2) SERVICE adds an explicit ownership guard in updateExpense mirroring the status guard so the API path cannot bypass the UI. (3) UI replaces the hardcoded isAccountsRole equals admin-or-accounts with a role-set constant matching RLS (resolving the EXP-012 divergence) and shows Edit only on the submitter own draft/pending/rejected rows. The mine/all toggle UI is deferred.

## migrationSql
```sql
DROP POLICY IF EXISTS expenses_select ON public.expenses;
CREATE POLICY expenses_select ON public.expenses
  AS PERMISSIVE FOR SELECT TO authenticated
  USING (
    is_staff_user()
    AND (
      created_by = auth.uid()
      OR has_role('manager')
      OR has_role('accounts')
      OR is_platform_admin()
    )
  );

DROP POLICY IF EXISTS expenses_update ON public.expenses;

CREATE POLICY expenses_update_approver ON public.expenses
  AS PERMISSIVE FOR UPDATE TO authenticated
  USING ( (SELECT has_role('accounts')) )
  WITH CHECK ( (SELECT has_role('accounts')) );

CREATE POLICY expenses_update_owner ON public.expenses
  AS PERMISSIVE FOR UPDATE TO authenticated
  USING ( created_by = auth.uid() AND status IN ('draft','pending','rejected') )
  WITH CHECK ( created_by = auth.uid() );
```

## VERDICT
fix (conf 0.8)

### issues
- Live DB confirms the design's policy baseline exactly (expenses_select USING(true); expenses_update USING+CHECK = has_role('accounts'); expenses_delete = has_role('admin') DELETE-only dead path; RESTRICTIVE tenant_isolation intact). created_by is nullable, 0 nulls, 2 rows, 0 approved/paid -> no backfill needed. These claims hold.
- BLAST RADIUS / shared path (not resolved): caseFinanceService.ts queries expenses directly (lines 53, 181, 190) for case profitability and getCaseExpenses. Under the tightened SELECT (is_staff_user AND created_by=auth.uid OR has_role('manager'/'accounts')), a technician who did not create an expense reads 0 rows -> silently understated case expense totals and case margin for the lab's core operator role. The design only flags this to 'VERIFY before applying' and does NOT resolve it.
- RLS self-approval backstop is WEAKENED, not strengthened. The new expenses_update_owner has WITH CHECK (created_by = auth.uid()) with NO status constraint. Because the owner and approver policies OR together, a creator can issue a raw UPDATE setting status='approved' on their own pending row -> the RLS no longer blocks creator self-approval (SoD now lives ONLY in approveExpense service code). The old single policy required has_role('accounts') on CHECK for every UPDATE, which did backstop this.
- Redundant/misleading predicate: has_role('accounts') already resolves to owner/admin/manager/accounts (verified function def), so 'has_role('manager') OR has_role('accounts')' in the SELECT scope is fully redundant. Harmless but the design mischaracterizes its own scope.
- is_staff_user() omits 'viewer' (verified) so viewers lose all expense reads -- this is the intended 3.7 fix, not a defect.
- Transitions are safe: approveExpense/rejectExpense/markExpenseAsPaid and record_expense_disbursement all authorize via has_role('accounts') server-side (verified RPC def), and the approver UPDATE policy preserves full has_role('accounts') power, so the UPDATE split does not break the state machine or the disbursement RPC.
- Service ownership guard design is sound and back-compatible (actingUserId optional, skips guard when omitted); UI ExpenseRow already carries created_by via EXPENSE_LIST_COLUMNS so the Edit-visibility gate has the data it needs.
- Does not duplicate shipped work: EXP-006 status guard and EXP-019 optimistic lock are preserved and untouched; this adds an orthogonal ownership layer.
- Session cost flagged CRITICAL at $581.79 by the cost hook -- operator should be aware before further DB introspection.

### mustFixBeforeShip
- Resolve technician case-expense visibility before tightening SELECT: either add `OR has_role('technician')` to the expenses_select scope, OR (preferred) expose case expense totals via a SECURITY DEFINER aggregate so caseFinanceService.getCaseFinancials/getCaseExpenses do not lose rows for technicians. Without this, case profitability (EXP-014, shipped) silently under-reports for the lab's primary operator role.
- Tighten expenses_update_owner WITH CHECK to forbid self-transition into approver-only states: add `AND status IN ('draft','pending','rejected')` to the WITH CHECK (not just USING), so a creator cannot raw-UPDATE their own row to status='approved'/'paid'. Keep SoD in approveExpense as defense-in-depth, but the RLS must not regress the creator-self-approval backstop the current single policy provides.
- Drop the redundant has_role('manager') term in the SELECT scope (subsumed by has_role('accounts')) or correct the design narrative to reflect that has_role('accounts') = owner/admin/manager/accounts.

### migrationSafety
Migration is additive and reversible in shape (DROP POLICY IF EXISTS + CREATE), touches only expenses_select and expenses_update, leaves RESTRICTIVE expenses_tenant_isolation and expenses_insert intact, needs no backfill (2 rows, 0 null created_by, 0 approved/paid). Lock risk is negligible (policy DDL on a 2-row table; brief ACCESS EXCLUSIVE on expenses only). BUT it is NOT behavior-safe as written: (1) the SELECT tightening is a live read regression for technicians on the shared caseFinanceService path; (2) the UPDATE split removes the RLS-level creator-self-approval backstop. Tenant isolation and append-only ledger/audit tables are untouched. Apply only after the two must-fix policy corrections; MAIN-LOOP-ONLY via mcp apply_migration, then regen database.types.ts (though no column change, the manifest/types pipeline should still run).
