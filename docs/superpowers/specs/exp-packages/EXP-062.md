# EXP-062 — Import never sets currency/exchange_rate/amount_base — imported expenses default to USD/1.0/NULL
priority=Critical verdict=fix conf=0.86 needsMigration=True risk=medium

## summary
The expense import path is a "dumb" batch INSERT (ImportWizard.tsx handleImport, lines 174-293) that builds records purely from mapped CSV columns and inserts straight into `expenses` via a loose-typed client. It never runs the rate-snapshot logic that `createExpense` (expensesService.ts:192-224) does, so imported rows fall to the live DB defaults verified this session: `currency 'USD'`, `exchange_rate 1`, `rate_source 'derived'`, and `amount_base`/`tax_amount_base` NULL. `getExpenseStats` (expensesService.ts:671-672) and `getExpensesByCategory` then treat every imported foreign-currency expense as already-base at rate 1 via the `amount_base ?? amount*exchange_rate` fallback, silently corrupting cross-currency totals for any non-base-currency lab. `ENTITY_CONFIGS.expenses` (importExportService.ts:160-175) also omits currency/exchange_rate/tax_amount/amount_base entirely and lists a phantom `paid_at` dateField (no such column — overlaps EXP-065).

The audit's verdict (line 746) is the spine of this design: adding config columns alone is NOT sufficient because the dumb INSERT still won't compute `amount_base`. The robust, write-path-agnostic fix is a BEFORE INSERT/UPDATE trigger on `expenses` that populates `amount_base`/`tax_amount_base` from `amount`/`tax_amount` × `exchange_rate` whenever they are NULL — so NO write path (import, future RPC, raw SQL, legacy backfill) can leave them unset. It mirrors the verified server-side helpers `_fin_base_currency(tenant)` and `_fin_currency_decimals(code)` and the `convertToBase = round(amount*rate, baseDecimals)` semantics already used by createExpense. Because createExpense/updateExpense already compute and pass `amount_base`, the trigger is a no-op for them (fills only when NULL) — zero behavior change on the form path. We then complete `ENTITY_CONFIGS.expenses` so importers can supply currency/exchange_rate/tax_amount, remove the phantom `paid_at`, and add a one-line in-wizard note. This is the SAFE, fully-additive slice; the larger "route import through an atomic approve+post RPC / force pending" ambition (EXP-031/063/064) is explicitly deferred.

## migrationSql
```sql
-- Migration: expenses_compute_base_amounts_trigger
-- EXP-062: Guarantee amount_base / tax_amount_base are populated on EVERY expense
-- write path (import, RPC, raw SQL, legacy backfill). The form path
-- (createExpense/updateExpense) already computes these; the trigger fills only
-- when NULL, so it is a no-op there and never overwrites a service-snapshotted base.
-- Mirrors convertToBase(amount, rate, baseDecimals) = round(amount*rate, base.dp)
-- using the existing server-side helpers _fin_base_currency / _fin_currency_decimals.

CREATE OR REPLACE FUNCTION public.set_expenses_base_amounts()
  RETURNS trigger
  LANGUAGE plpgsql
  SECURITY DEFINER
  SET search_path = public
AS $$
DECLARE
  v_base text;
  v_dec  integer;
  v_rate numeric;
BEGIN
  -- tenant_id is already set by set_expenses_tenant_and_audit (fires first; this
  -- trigger is named zz_* so it sorts last among BEFORE row triggers).
  v_base := public._fin_base_currency(NEW.tenant_id);
  v_dec  := public._fin_currency_decimals(v_base);
  v_rate := COALESCE(NEW.exchange_rate, 1);

  IF NEW.amount_base IS NULL THEN
    NEW.amount_base := round(COALESCE(NEW.amount, 0) * v_rate, v_dec);
  END IF;

  IF NEW.tax_amount_base IS NULL THEN
    NEW.tax_amount_base := round(COALESCE(NEW.tax_amount, 0) * v_rate, v_dec);
  END IF;

  RETURN NEW;
END;
$$;

REVOKE ALL ON FUNCTION public.set_expenses_base_amounts() FROM public;

DROP TRIGGER IF EXISTS zz_set_expenses_base_amounts ON public.expenses;
CREATE TRIGGER zz_set_expenses_base_amounts
  BEFORE INSERT OR UPDATE ON public.expenses
  FOR EACH ROW
  EXECUTE FUNCTION public.set_expenses_base_amounts();

-- One-time additive backfill of pre-existing NULL base rows (no hard delete).
UPDATE public.expenses e
SET amount_base = round(COALESCE(e.amount,0) * COALESCE(e.exchange_rate,1),
                        public._fin_currency_decimals(public._fin_base_currency(e.tenant_id))),
    tax_amount_base = round(COALESCE(e.tax_amount,0) * COALESCE(e.exchange_rate,1),
                        public._fin_currency_decimals(public._fin_base_currency(e.tenant_id)))
WHERE (e.amount_base IS NULL OR e.tax_amount_base IS NULL)
  AND e.deleted_at IS NULL;
```

## blastRadius
The BEFORE trigger fires on EVERY expenses INSERT/UPDATE, so the blast surface is the entire expense write path — which is why fill-only-when-NULL is the load-bearing guard: createExpense/updateExpense compute and pass amount_base/tax_amount_base non-NULL, making the trigger a verified no-op there (no double conversion, no clobbering of a manually overridden rate). The only rows it changes are those written without base values (imports, raw SQL, legacy/backfill). Trigger ordering is the second risk: it MUST run after set_expenses_tenant_and_audit so NEW.tenant_id is set before _fin_base_currency(NEW.tenant_id); guaranteed by the zz_ name via Postgres alphabetical same-timing ordering. If tenant_id were ever NULL, _fin_base_currency safely falls back to 'USD'->2dp rather than erroring. Reports protected (all read amount_base): getExpenseStats, getExpensesByCategory, financialReportsService P&L/cash-flow/revenue-by-case — once imports carry correct base, their amount_base ?? amount*rate fallbacks stop firing for imports. NOT touched: financial_transactions ledger (append-only; import still posts no ledger row — that gap is EXP-031, deferred), vat_records, invoices/sales VAT (calculateVATForPeriod in vatService.ts is untouched — EXP-032 stays deferred), banking. exchange_rate=0 (EXP-020) would also zero amount_base via the trigger, but that pre-exists and is a separate finding; the trigger does not worsen it. The migration is function+trigger only (no column shape change) but CI still requires database.types.ts regen + manifest entry (schema-drift / migration-manifest gates).

### issues
- FALSE CENTRAL CLAIM: paid_at is NOT a phantom column. Live schema confirms expenses.paid_at exists (timestamp with time zone, nullable). The design's instruction to set dateFields -> ['expense_date','approved_at'] (REMOVE paid_at) is a regression: it drops the ability to import the paid timestamp and contradicts the design's own EXP-065 dependency note. paid_at should be KEPT.
- Trigger is a complete guarantee only for INSERT and NULL-base rows. fill-only-when-NULL means a raw UPDATE that changes amount without nulling amount_base leaves a stale base. The summary's framing ('NO write path can leave them unset') overstates the guarantee; the real win is the import INSERT path + NULL-base legacy rows.
- exchange_rate=0 sharp edge: a literal '0' rate in a CSV passes through and the trigger computes amount_base = amount*0 = 0. Deferred to EXP-020 but not guarded here. A blank rate is safer (parseFloat -> NaN -> field skipped -> DB default exchange_rate=1).
- Verified-correct (no fix): _fin_base_currency(p_tenant uuid)->text and _fin_currency_decimals(p_code text)->integer exist as SECURITY DEFINER; trigger ordering via zz_ prefix is correct (set_tenant_and_audit_fields sets tenant_id when NULL on INSERT, fires first alphabetically); createExpense computes non-NULL base so trigger is a no-op there; getExpenseStats fallback at lines 671-672 matches exactly; stringFields columns currency/vendor/reference/notes all exist; dropdown union at line 605 is the correct anchor.
- ImportWizard empty-value guard (line 210) drops blank currency cells -> falls to DB default 'USD' silently, which is the exact corruption mode for a non-USD-base lab. Advisory is guidance-only, not enforcement; full enforcement correctly deferred to EXP-031/063/064.
- Migration is additive/reversible; backfill touches only 2 live rows (0 NULL base, 0 foreign ccy) -> negligible lock risk. RLS/tenant isolation intact. financial_transactions, vat_records, sales/invoice VAT untouched.

### mustFixBeforeShip
- Do NOT remove paid_at from ENTITY_CONFIGS.expenses.dateFields. paid_at is a real column (verified live). Keep dateFields = ['expense_date','approved_at','paid_at']. Fix the design narrative that calls it phantom and reconcile the EXP-065 dependency note (its premise is wrong).
- Invert the test's config-shape guard: it currently asserts dateFields EXCLUDES 'paid_at' — that locks in the regression. Assert dateFields INCLUDES 'paid_at' (plus expense_date, approved_at). Keep the numberFields/stringFields assertions.
- Scope the migration comment honestly: change 'no write path can leave them unset' to state the trigger fills on INSERT and for NULL-base rows but does NOT recompute base on an UPDATE that mutates amount while amount_base is already non-NULL — so EXP-031 builders don't over-trust the net on the UPDATE path.
