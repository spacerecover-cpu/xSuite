# EXP-064 — Import doesn't stamp created_by and never validates category_id/case_id existence/tenancy; whole-batch failures
priority=High verdict=fix conf=0.82 needsMigration=True risk=medium

## summary
Grounded in the live code/DB, EXP-064 has three independent defects, each fixed at the right layer:

1) created_by is NULL on imported expenses. The only BEFORE trigger on `expenses` is `set_expenses_tenant_and_audit` → `set_tenant_and_audit_fields()`, which stamps tenant_id/created_at/updated_at but NOT created_by (verified via pg_get_functiondef). The CLAUDE.md note that `set_audit_actor_fields()` covers expenses is inaccurate — that trigger is NOT attached to `expenses`. The ImportWizard also never sends created_by (records are built solely from `fieldMappings` in handleImport, ImportWizard.tsx:196-243). FIX (DB, robust, single chokepoint): add a BEFORE INSERT/UPDATE trigger `set_expenses_audit_actor` running a SECURITY DEFINER fn that does `NEW.created_by := COALESCE(NEW.created_by, auth.uid())` on INSERT and `NEW.updated_by := COALESCE(auth.uid(), NEW.updated_by)` on UPDATE. This makes EVERY write path (import, form createExpense, future RPCs) attribute the submitter without touching any caller. (Aligns with PR-7's EXP-010 actor-stamping intent; I scope it to the additive trigger only so it is safe to ship inside the import PR.)

2) No reference existence/tenancy validation. `validateField` (importExportService.ts:888-1005) never checks `referenceFields`, so a bogus/wrong-tenant category_id/case_id passes and FK-fails at insert. master_expense_categories is GLOBAL (no tenant_id; has is_active, no deleted_at). cases is TENANT-SCOPED (has deleted_at; RLS already restricts SELECT to current tenant). FIX (frontend, async): add `validateExpenseReferences(rows, mappings)` to importExportService that batches DISTINCT mapped category_id/case_id values and checks them — category_id against master_expense_categories (exists AND is_active), case_id against cases (exists AND deleted_at IS NULL; RLS makes a wrong-tenant id resolve to "not found", so this doubles as the tenancy check). Wire it into ImportWizard.handleValidate so bad references become per-row validation errors (excluded from import, listed in the errors panel/CSV) BEFORE any insert. Mirrors the existing bulkLookupNames batching pattern.

3) Whole-batch failure on one bad FK. handleImport does `errorCount += batch.length` on any batch error (ImportWizard.tsx:272). FIX (frontend): keep the fast bulk insert as the happy path, but on batch error fall back to per-row inserts for that batch, counting successes/failures per row and capturing each failed row's DB message into importResult so 99 good rows still land and the operator sees which row failed.

Safe-slice boundaries (explicitly deferred to their own findings, not regressed here): currency/exchange_rate/amount_base snapshot on import = EXP-062 (separate); ledger/VAT posting on import = EXP-031; bulk_import/usage gating = EXP-063; phantom paid_at + expense_number unique index = EXP-065; import job/log provenance rows = EXP-067; number/date coercion = EXP-068. This package does NOT route import through createExpense (that is EXP-031/EXP-062's atomic-RPC scope) to keep blast radius contained.

## migrationSql
```sql
-- Migration: expenses_audit_actor_stamping
-- Stamps created_by/updated_by on every expenses write so no path (import, form, RPC) leaves attribution NULL.
CREATE OR REPLACE FUNCTION public.set_expenses_audit_actor()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path TO 'public'
AS $function$
BEGIN
  IF TG_OP = 'INSERT' THEN
    NEW.created_by := COALESCE(NEW.created_by, auth.uid());
  ELSIF TG_OP = 'UPDATE' THEN
    -- Preserve prior editor when there is no auth context (system/bypass jobs).
    NEW.updated_by := COALESCE(auth.uid(), NEW.updated_by);
  END IF;
  RETURN NEW;
END;
$function$;

DROP TRIGGER IF EXISTS set_expenses_audit_actor ON public.expenses;
CREATE TRIGGER set_expenses_audit_actor
  BEFORE INSERT OR UPDATE ON public.expenses
  FOR EACH ROW EXECUTE FUNCTION public.set_expenses_audit_actor();

-- No data backfill: historical rows with created_by IS NULL are left unchanged
-- (no reliable actor to attribute retroactively). The existing
-- set_expenses_tenant_and_audit trigger is intentionally left in place.
```

## blastRadius
What could break and how it's avoided: (1) The new created_by trigger fires on ALL expenses writes, including the form createExpense path. It only sets created_by when NULL (COALESCE), so the form path — which doesn't pass created_by — gains correct attribution and existing rows that DO pass it are untouched. updated_by uses COALESCE(auth.uid(), NEW.updated_by) so system/bypass updates don't wipe it. Both this trigger and set_expenses_tenant_and_audit are BEFORE-row and idempotent, so coexistence is safe regardless of fire order. No invoice/VAT/banking/report path touches expenses triggers. (2) Reference validation is gated on entityType==='expenses' and runs only in the wizard's validate step — zero effect on cases/invoices/inventory/other imports or on any export path. The cases lookup relies on existing RLS for tenancy (no new policy), so it cannot leak cross-tenant rows. (3) Per-row fallback only activates when a bulk batch ALREADY errored (today that whole batch is lost anyway), so clean imports and all other entity types keep the fast single bulk insert — no performance regression and no behavior change on the success path. (4) financial_transactions is untouched (no ledger posting added here — deferred to EXP-031), so the append-only/reverse-only invariant is not engaged. (5) No VAT path touched, so EXP-032's shared calculateVATForPeriod / sales vat_records risk is entirely out of scope for this item. (6) Soft-delete respected: case_id validation filters deleted_at IS NULL; no hard deletes introduced.
