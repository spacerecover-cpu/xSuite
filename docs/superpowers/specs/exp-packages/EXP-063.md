# EXP-063 — Bulk import bypasses max_expenses_per_month and bulk_import plan gates (UI-only enforcement)
priority=High verdict=fix conf=0.82 needsMigration=True risk=medium

## summary
The expense bulk-import path has zero feature/usage enforcement. The ImportWizard (`src/components/importExport/ImportWizard.tsx` handleImport, lines 174-293) does a direct `supabase.from('expenses').insert(batch)` per 100 rows and never calls hasFeature/checkUsageLimit/canPerformAction. The only gate that exists is the UI-only `UsageLimitGuard` wrapping the create button in `ExpenseFormModal.tsx:427`. Live DB confirms `expenses_insert` RLS = `has_role('accounts')` only (no plan check), no `create_expense`/limit RPC exists, and there are ZERO `plan_features` rows for `bulk_import` or `max_expenses_per_month` — so today the monthly cap resolves to null (unlimited) and `bulk_import` resolves only via the client-side FEATURE_REQUIREMENTS hierarchy (requires 'professional'). A Starter tenant can load thousands of expenses via Import, bypassing both gates and any future per-month cap.

Durable fix = a single server-side chokepoint, additive and independent of EXP-031 (which is not shipped). Add one SECURITY DEFINER RPC, `assert_expense_import_allowed(p_row_count int)`, that: (1) authorizes via `has_role('accounts')` (RLS is bypassed under definer); (2) resolves the caller's active plan + the `bulk_import` feature flag + the `max_expenses_per_month` limit from `tenant_subscriptions`/`plan_features` using the SAME resolution rules the client uses (explicit plan_features row wins; else feature-hierarchy fallback for bulk_import requiring 'professional'; else null=unlimited for the limit); (3) counts this tenant's current-calendar-month expenses (NOT deleted) and RAISES EXCEPTION with a stable error code/message if either the feature is missing or current+p_row_count would exceed the limit. The ImportWizard calls this RPC once before the insert loop and aborts the whole import on failure. The ImportExport page additionally hides/disables the Expenses Import entry behind `hasFeature('bulk_import')` for a clean UX (defense-in-depth, not the authority). This is the shared server-side enforcement the audit calls out for EXP-063/EXP-071; it does not touch the create/edit/approve write paths, invoices, VAT, banking, or reports.

Note: import is generic across ~12 entity types but only Expenses has these specific gates; the RPC and the UI gate are expense-scoped, so other entity imports are untouched.

## migrationSql
```sql
-- EXP-063: server-side enforcement of bulk_import feature + max_expenses_per_month quota on expense bulk import.
-- SECURITY DEFINER so it can read tenant_subscriptions/plan_features regardless of caller RLS; authorize explicitly inside.
-- Mirrors the client resolution in featureGateService.ts: explicit plan_features row wins; else hierarchy fallback for
-- bulk_import (requires 'professional'); else NULL = unlimited for max_expenses_per_month. Counts current calendar-month
-- non-deleted expenses for the caller's tenant.

CREATE OR REPLACE FUNCTION public.assert_expense_import_allowed(p_row_count integer)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_tenant_id   uuid;
  v_plan_code   text;
  v_plan_id     uuid;
  v_plan_rank   integer;
  v_bulk_row    record;       -- plan_features row for bulk_import (may be missing)
  v_bulk_ok     boolean;
  v_limit       integer;      -- max_expenses_per_month limit_value (NULL = unlimited)
  v_limit_found boolean := false;
  v_current     integer;
  v_month_start timestamptz;
BEGIN
  -- Authorize: only accounts-capable staff may bulk-import expenses (same gate as expenses_insert RLS).
  IF NOT has_role('accounts') THEN
    RAISE EXCEPTION 'Not authorized to import expenses'
      USING ERRCODE = '42501';
  END IF;

  IF p_row_count IS NULL OR p_row_count < 0 THEN
    RAISE EXCEPTION 'Invalid import row count'
      USING ERRCODE = '22023';
  END IF;

  v_tenant_id := get_current_tenant_id();
  IF v_tenant_id IS NULL THEN
    -- Platform admin / no-tenant context: nothing to meter, allow.
    RETURN;
  END IF;

  -- Resolve active plan code (default 'starter' when no active subscription), mirroring loadPlanCache().
  SELECT ts.plan_id, sp.code
    INTO v_plan_id, v_plan_code
  FROM tenant_subscriptions ts
  JOIN subscription_plans sp ON sp.id = ts.plan_id
  WHERE ts.tenant_id = v_tenant_id
    AND ts.status = 'active'
    AND ts.deleted_at IS NULL
  ORDER BY ts.created_at DESC
  LIMIT 1;

  v_plan_code := COALESCE(v_plan_code, 'starter');
  v_plan_rank := CASE v_plan_code
                   WHEN 'starter' THEN 1
                   WHEN 'professional' THEN 2
                   WHEN 'enterprise' THEN 3
                   ELSE 0
                 END;

  -- ---- bulk_import feature gate ----
  IF v_plan_id IS NOT NULL THEN
    SELECT * INTO v_bulk_row
    FROM plan_features
    WHERE plan_id = v_plan_id
      AND feature_key = 'bulk_import'
      AND deleted_at IS NULL
    LIMIT 1;
  END IF;

  IF FOUND THEN
    v_bulk_ok := COALESCE(v_bulk_row.is_enabled, false);
  ELSE
    -- Hierarchy fallback: bulk_import requires 'professional' (rank 2).
    v_bulk_ok := (v_plan_rank >= 2);
  END IF;

  IF NOT v_bulk_ok THEN
    RAISE EXCEPTION 'Bulk import requires the Professional plan or higher. Please upgrade to import expenses.'
      USING ERRCODE = 'P0001', HINT = 'feature:bulk_import';
  END IF;

  -- ---- max_expenses_per_month quota ----
  IF v_plan_id IS NOT NULL THEN
    SELECT limit_value INTO v_limit
    FROM plan_features
    WHERE plan_id = v_plan_id
      AND feature_key = 'max_expenses_per_month'
      AND deleted_at IS NULL
    LIMIT 1;
    v_limit_found := FOUND;
  END IF;

  -- No explicit limit row => unlimited (matches client: limit = null).
  IF NOT v_limit_found OR v_limit IS NULL THEN
    RETURN;
  END IF;

  v_month_start := date_trunc('month', now());

  SELECT count(*) INTO v_current
  FROM expenses
  WHERE tenant_id = v_tenant_id
    AND deleted_at IS NULL
    AND created_at >= v_month_start;

  IF (v_current + p_row_count) > v_limit THEN
    RAISE EXCEPTION 'This import of % expense(s) would exceed your plan limit of % per month (% already used this month). Please upgrade or reduce the file.',
      p_row_count, v_limit, v_current
      USING ERRCODE = 'P0001', HINT = 'limit:max_expenses_per_month';
  END IF;
END;
$$;

REVOKE ALL ON FUNCTION public.assert_expense_import_allowed(integer) FROM public;
GRANT EXECUTE ON FUNCTION public.assert_expense_import_allowed(integer) TO authenticated;

COMMENT ON FUNCTION public.assert_expense_import_allowed(integer) IS
  'EXP-063: server-side gate for expense bulk import. Authorizes via has_role(accounts); enforces bulk_import feature + max_expenses_per_month quota for the caller tenant; raises on denial.';
```

## blastRadius

The new RPC is additive (no signature/behavior change to any existing function) and is invoked only from the expense branch of the import wizard, so invoices, VAT, banking, payments, reports, and other entity imports are untouched — verified that the import insert path and the financial write paths are separate. RLS is unchanged (expenses_insert stays has_role('accounts')); the RPC adds enforcement on top, it does not relax anything. Behavioral-parity risk: the audit's live-DB reality is that plan_features has NO rows for these two keys, so today the gate is effectively open (bulk_import via hierarchy=professional, limit=unlimited). The RPC deliberately mirrors that exact resolution (row-wins, else hierarchy, else unlimited) so it will NOT suddenly start blocking tenants who weren't blocked before — it only enforces once operators seed plan_features (which is the intended billing config). One subtle divergence from the client by design: the server counts NON-deleted expenses (`deleted_at IS NULL`), whereas the client checkUsageLimit('max_expenses_per_month') currently counts ALL rows including soft-deleted; the server is the more correct metering basis and is the authority — flag this so the client count can be aligned later (not in this PR, to keep blast radius small). Failure mode is safe: if the RPC errors for any reason the wizard aborts the import (no partial silent inserts). The only user-visible change for a correctly-entitled tenant is none; for an under-entitled tenant the import is now refused with a clear upgrade message instead of silently succeeding.

### issues
- NOT true insert-path enforcement: the RPC is a wizard preflight only. expenses_insert RLS stays has_role('accounts') with no plan check, so a direct supabase-js/API caller (or any non-wizard insert path) bypasses the gate entirely. The summary overstates this as 'server-side enforcement'/'the authority' — it is server-side COMPUTATION invoked from one client path, not a constraint on writes. Acceptable for the stated scope but must be labeled honestly.
- Non-atomic / TOCTOU: gate checks validRows.length once before the batch loop; concurrent imports or the partial-success loop (batch errors increment errorCount but the loop keeps inserting subsequent batches) can push the month count past the quota. Soft gate only.
- Resolution divergence from client (by design but undocumented in code): RPC filters tenant_subscriptions.deleted_at IS NULL and plan_features.deleted_at IS NULL; client loadPlanCache (featureGateService.ts L89-100, L112-115) filters NEITHER — it keys only on status='active' and plan_id. Zero impact today (0 rows) but the two will diverge once soft-deleted plan rows exist.
- Month-boundary timezone mismatch: client computes startOfMonth in BROWSER-LOCAL time (checkUsageLimit L262-264) then toISOString(); RPC uses date_trunc('month', now()) in server UTC. For GMT+4 tenants these differ by up to ~a day at month edges, so client preview and server gate can disagree on the boundary. Cosmetic for metering.
- ImportExport.tsx 'optional' suggestion to add disabled={!canBulkImport} on the Import button (L186-193) is WRONG as written: that button is shared across all ~12 entity types in the .map(), so it would disable Import for customers/invoices/etc., not just expenses. Only the handleEntitySelect guard (L61-68) is correctly expense-scoped — but note that guard as proposed ('if (!canBulkImport) return') also blocks ALL entities, not just expenses; it must additionally check entityType==='expenses'.
- Server counts deleted_at IS NULL expenses; client checkUsageLimit('max_expenses_per_month') counts ALL rows incl. soft-deleted (no deleted_at filter, L266-270). Design flags this as a deliberate, deferred divergence — acceptable but the client preflight message can disagree with the server.
- tsc will break until types regen: supabase.rpc('assert_expense_import_allowed', ...) is not in the Functions map yet. Design acknowledges this; main loop must regen database.types.ts via MCP after applying the migration, or the typecheck CI gate fails.
