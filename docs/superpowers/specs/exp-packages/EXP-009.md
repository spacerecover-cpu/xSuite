# EXP-009 — No DB-side state-machine trigger backstop on expense transitions
priority=High verdict=fix conf=0.86 needsMigration=True risk=medium

## summary
EXP-009's ideal had two halves: (1) an allowed-transition map enforced in the service, and (2) a DB-side trigger backstop so the invariant holds even when a privileged user calls the table UPDATE directly (RLS expenses_update = has_role('accounts'), no status predicate). In the CURRENT code, half (1) is already shipped (#310-#317): every transition in src/lib/expensesService.ts reads current status and uses .eq('status', ...) guards (approveExpense requires pending + creator!=approver; rejectExpense requires pending; markExpenseAsPaid and record_expense_disbursement require approved; updateExpense blocks approved/paid/voided; submitExpense uses .in(['draft','rejected']) without overwriting created_by; deleteExpense sets voided). The remaining gap is purely the trigger backstop: an accounts-role user can still run supabase.from('expenses').update({status:'paid'}).eq('id',E) on a draft, or flip an approved-and-ledgered row to rejected (dangling financial_transactions posting), bypassing the service. This adds a BEFORE UPDATE trigger enforce_expense_status_transition() (SECURITY DEFINER, SET search_path=public) validating OLD.status->NEW.status against a fixed allowed-edge set, raising check_violation on illegal edges. It allows exactly the edges the shipped service paths use, so it breaks nothing. No new columns; the live status CHECK already lists all six states. Live data is 1 draft + 1 pending, so it is non-breaking.</summary>
<parameter name="migrationSql">CREATE OR REPLACE FUNCTION public.enforce_expense_status_transition()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path TO 'public'
AS $function$
DECLARE
  old_s text := OLD.status;
  new_s text := NEW.status;
  soft_deleting boolean := (OLD.deleted_at IS NULL AND NEW.deleted_at IS NOT NULL);
BEGIN
  -- Fast path: status unchanged -> nothing to validate (field edits, timestamp/audit bumps).
  IF new_s IS NOT DISTINCT FROM old_s THEN
    RETURN NEW;
  END IF;

  -- ->voided only as part of a soft delete (deleteExpense sets status='voided' + deleted_at).
  IF new_s = 'voided' THEN
    IF soft_deleting THEN
      RETURN NEW;
    END IF;
    RAISE EXCEPTION 'An expense can only become voided via soft-delete (% -> voided without deleted_at)', old_s
      USING ERRCODE = 'check_violation';
  END IF;

  -- voided is terminal.
  IF old_s = 'voided' THEN
    RAISE EXCEPTION 'A voided expense cannot transition to %', new_s
      USING ERRCODE = 'check_violation';
  END IF;

  -- Allowed forward edges (mirror the shipped service guards exactly).
  IF (old_s IN ('draft','rejected') AND new_s = 'pending')
     OR (old_s = 'pending'  AND new_s = 'approved')
     OR (old_s = 'pending'  AND new_s = 'rejected')
     OR (old_s = 'approved' AND new_s = 'paid')
  THEN
    RETURN NEW;
  END IF;

  RAISE EXCEPTION 'Illegal expense status transition: % -> %', old_s, new_s
    USING ERRCODE = 'check_violation';
END;
$function$;

REVOKE ALL ON FUNCTION public.enforce_expense_status_transition() FROM PUBLIC;

DROP TRIGGER IF EXISTS zz_enforce_expense_status_transition ON public.expenses;
CREATE TRIGGER zz_enforce_expense_status_transition
  BEFORE UPDATE ON public.expenses
  FOR EACH ROW
  WHEN (
    OLD.status IS DISTINCT FROM NEW.status
    OR (OLD.deleted_at IS NULL AND NEW.deleted_at IS NOT NULL)
  )
  EXECUTE FUNCTION public.enforce_expense_status_transition();

## blastRadius
Why it does not break adjacent systems: (1) record_expense_disbursement (banking) does approved->paid by direct UPDATE - explicitly allowed; verified against its live definition. (2) deleteExpense sets status='voided' + deleted_at from ANY state - the soft-delete branch allows this, so archiving approved/paid/draft expenses and the in-function ledger reversal (reverse_financial_transaction) are untouched. (3) The existing BEFORE trigger set_expenses_tenant_and_audit is unaffected; the new trigger is named zz_ so it runs after it and validates the final row. (4) Field-only edits do not change status, so the WHEN clause skips the trigger entirely - zero overhead/behaviour change on the edit hot path. (5) Imports/CSV (importExportService, ImportWizard) insert at DEFAULT 'draft' and never UPDATE status -> unaffected. Out of scope and safe: invoices/quotes/payments/VAT/reports/banking balances - trigger is scoped to public.expenses only; financial_transactions stays append-only, reversed only via the existing RPC inside deleteExpense (unchanged). Residual risk (why medium): any undiscovered legitimate status UPDATE outside the five mapped edges would now be rejected. Mitigated by grepping .update on 'expenses' with a status key (only the service mutators + record_expense_disbursement exist) and by the rolled-back live verification before merge. The map is intentionally minimal; a future legitimate edge (e.g. paid->refunded) is a one-line additive change.
