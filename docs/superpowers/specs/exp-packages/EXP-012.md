# EXP-012 — UI approver gate (admin||accounts) diverges from RLS (owner/admin/manager/accounts); hardcoded role strings
priority=High verdict=ship conf=0.86 needsMigration=False risk=medium

## summary
EXP-012 is a frontend-only authorization-divergence fix. The live RLS for the expense mutate path is `expenses_update` / `expenses_insert` = `has_role('accounts')`, which (per the live `has_role` definition I read) resolves to the hierarchical set {owner, admin, manager, accounts}. But `ExpensesList.tsx` hardcodes `isAccountsRole = profile?.role === 'admin' || profile?.role === 'accounts'` (line 126) and `canBulkArchive = profile?.role === 'owner' || profile?.role === 'admin'` (line 96), gating the Approve/Reject (line 675), Mark-as-Paid (line 693) affordances. Consequences observed in code: (1) owner and manager users — who CAN approve/reject/mark-paid per RLS and via the service functions directly — never see those buttons, so the on-screen control list is untrustworthy and they have hidden powers; (2) the component uses raw role-string literals instead of a permissions abstraction, which CLAUDE.md forbids ("Never hardcode role strings in components — use the permission system").

The safe, correct scope: introduce ONE canonical client-side helper that mirrors the `has_role('accounts')` server set — {owner, admin, manager, accounts} — and use it to gate the approver affordances; converge `isAccountsRole` to that set. This makes the UI exactly match what RLS permits, so the buttons no longer lie. The SoD guard (creator ≠ approver) already lives server-side in approveExpense (lines 402-405, EXP-008, shipped), so broadening the visible approver set introduces NO new self-approval risk: an owner/manager who created the expense will still be rejected by the service with "You cannot approve an expense you created", surfaced via the existing onError toast. No new approval power is granted that RLS+service don't already allow — this only makes the affordance visible to users who already had the capability.

Bulk-archive (`canBulkArchive`, owner/admin) is intentionally NARROWER than the soft-delete UPDATE RLS and is the subject of a separate finding (EXP-018, which wants archive TIGHTENED to admin via RLS/RPC, not broadened). Therefore EXP-012's scope explicitly EXCLUDES touching canBulkArchive — changing it here would collide with EXP-018's intended direction. I replace only the literal it's built from with the shared helper's building blocks where it improves clarity, but keep its role set unchanged.

Grounding: I read the live RLS policies on public.expenses, the live has_role(text) function body, expensesService.ts (approve/reject/markPaid all already status- and SoD-guarded server-side), ExpensesList.tsx (the two hardcoded gates), PermissionsContext.tsx (module-level only — no per-action 'approve expense' key exists, confirmed by memory obs 8876), and roles.ts (note: roles.ts omits 'manager' from its ROLES tuple even though the live role hierarchy and profiles include it — so a typed helper must accept the broader string set, not be keyed to roles.ts's Role type).

## files
### create `src/lib/roleGates.ts`
New tiny shared module exporting the canonical client-side mirror of the server has_role sets, so no component hardcodes role literals (CLAUDE.md compliance). Content outline:

  import type { Database } from '../types/database.types';
  // Profiles.role is the source of truth for the string set; do NOT key off roles.ts Role (it omits 'manager').
  type ProfileRole = NonNullable<Database['public']['Tables']['profiles']['Row']['role']>;

  // Mirrors the server has_role('accounts') hierarchy: owner/admin/manager/accounts.
  // This is the role set RLS authorizes to insert/update/approve/reject expenses.
  export const EXPENSE_APPROVER_ROLES = ['owner', 'admin', 'manager', 'accounts'] as const;

  export function canManageExpenses(role: string | null | undefined): boolean {
    return !!role && (EXPENSE_APPROVER_ROLES as readonly string[]).includes(role);
  }

Keep it dependency-free and documented as 'the client mirror of has_role(accounts); update both together if the policy changes.' If a future generic role-hierarchy helper is wanted it can subsume this, but scope here is expenses only.

## blastRadius
Scope is the Expenses list page only. What could break and why it won't: (1) Invoices/VAT/banking/reports are untouched — no service, RPC, RLS, financial_transactions, vat_records, or report code changes; the only logic change is which roles SEE three buttons. (2) Granting visibility to owner/manager cannot post bad ledger/VAT rows: approveExpense already guards status==='pending' AND created_by!==approvedBy server-side, and the ledger/VAT writes are downstream of those guards — a newly-visible owner/manager clicking Approve on their own expense hits the existing 'You cannot approve an expense you created' throw, surfaced by the onError toast (line 238). (3) Mark-as-Paid (recordExpenseDisbursement, EXP-017) is an atomic RPC with its own approved->paid state + insufficient-funds guards, so widening its button visibility to owner/manager — already permitted by RLS — adds no unguarded cash path. (4) No new tokens/colors; the new title='Approve Expense' is a string only. (5) The canonical-set drift risk (UI says owner/admin/manager/accounts; RLS later changes) is mitigated by the parity test asserting the exact array and the doc-comment telling maintainers to update has_role and roleGates.ts together. Net: this strictly REDUCES divergence between UI and RLS; it does not expand the actual permission surface (RLS+service are the real gate and are unchanged).
