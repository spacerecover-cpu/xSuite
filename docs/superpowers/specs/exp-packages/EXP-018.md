# EXP-018 — Delete authority incoherent: bulk-archive (owner/admin) vs soft-delete-by-UPDATE (accounts+); DELETE policy dead
priority=High verdict=fix conf=0.82 needsMigration=True risk=medium

## summary
Today the "delete-equivalent" for an expense is a `deleted_at` UPDATE, governed by `expenses_update` RLS = `has_role('accounts')` (owner/admin/manager/accounts) — verified live. The dedicated `expenses_delete` policy = `has_role('admin')` governs only hard DELETE, which the app never issues, so it is dead. Two app paths exist and disagree: (1) `deleteExpense(id)` in expensesService.ts (lines 315-357) soft-deletes via UPDATE and already reverses the ledger via `reverse_financial_transaction`, retires VAT rows, cascades attachment soft-delete, and sets status='voided' for approved/paid — but it has ZERO UI callers (confirmed by grep; only the barrel export references it). (2) The actual UI delete is the inline `handleBulkArchive` in ExpensesList.tsx (lines 368-400), a raw `supabase.from('expenses').update({deleted_at})` gated UI-ONLY by `canBulkArchive = role in (owner,admin)` (line 96) — with NO ledger reversal and NO VAT retirement, so bulk-archiving an approved/ledgered expense orphans its GL row and VAT. The UI message says "Only admins can bulk archive" yet RLS lets manager/accounts do the same via the API/single-row path.

The fix converges all three (UI bulk, single-row service, RLS) onto ONE canonical authority = admin, by introducing a SECURITY DEFINER `archive_expense(p_expense_id uuid, p_reason text)` RPC modelled exactly on the existing `record_expense_disbursement` RPC (tenant guard, `has_role('admin')` authorization inside the definer since RLS is bypassed, `FOR UPDATE` lock, idempotent already-deleted guard). The RPC performs the full lifecycle atomically server-side: for approved/paid expenses it reverses every linked `financial_transactions` row via `reverse_financial_transaction`, soft-deletes the matching `vat_records`, cascades `expense_attachments` soft-delete, then sets `deleted_at=now()` + `status='voided'`. The `deleted_at` UPDATE is gated to admin via a column-aware BEFORE UPDATE trigger guard that fires only on the NULL->non-NULL transition (cleaner than splitting the policy, because RLS WITH CHECK cannot reference OLD). Both `deleteExpense` (rewritten to call the RPC) and `handleBulkArchive` (rewritten to loop the RPC) become thin wrappers; the UI gate stays admin and is now backed by RLS+trigger instead of being cosmetic.

Net: one source of truth for "who can remove a financial record" (admin), one code path that always reverses the ledger/VAT, and the previously-dead admin gate becomes the enforced authority for the real delete-equivalent. Reuses the EXP-006 reversal logic already shipped in #311.

## migrationSql

```sql
-- Migration: expense_archive_authority (EXP-018)
-- 1) Column-aware soft-delete guard: setting deleted_at on an expense (the real
--    delete-equivalent) requires admin, matching the intent of the dead expenses_delete
--    policy. RLS WITH CHECK can't see OLD, so enforce the NULL->non-NULL transition in a
--    BEFORE UPDATE trigger. Reversal/voiding flows through it because the RPC runs as a
--    SECURITY DEFINER whose session user is admin-checked before the UPDATE.

CREATE OR REPLACE FUNCTION public.enforce_expense_archive_authority()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path TO 'public'
AS $$
BEGIN
  -- Only police the transition into archived/soft-deleted state.
  IF NEW.deleted_at IS NOT NULL AND OLD.deleted_at IS NULL THEN
    IF NOT (has_role('admin') OR is_platform_admin()) THEN
      RAISE EXCEPTION 'Only an admin can archive (delete) an expense';
    END IF;
  END IF;
  RETURN NEW;
END;
$$;

DROP TRIGGER IF EXISTS trg_enforce_expense_archive_authority ON public.expenses;
CREATE TRIGGER trg_enforce_expense_archive_authority
  BEFORE UPDATE ON public.expenses
  FOR EACH ROW
  EXECUTE FUNCTION public.enforce_expense_archive_authority();

-- 2) Canonical archival RPC: the single soft-delete path. Mirrors
--    record_expense_disbursement (tenant guard, admin authz inside the definer,
--    FOR UPDATE lock, idempotent guard) and folds in the EXP-006 reversal logic so
--    bulk and single-row delete can never again orphan the GL/VAT.

CREATE OR REPLACE FUNCTION public.archive_expense(p_expense_id uuid, p_reason text DEFAULT NULL::text)
RETURNS public.expenses
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path TO 'public'
AS $$
DECLARE
  v_tenant uuid := get_current_tenant_id();
  v_exp    public.expenses;
  v_txn    record;
BEGIN
  IF v_tenant IS NULL THEN
    RAISE EXCEPTION 'No tenant context';
  END IF;

  -- Authorization (RLS bypassed under SECURITY DEFINER): admin is the canonical
  -- authority for removing a financial record (converges UI + RLS intent, EXP-018).
  IF NOT (has_role('admin') OR is_platform_admin()) THEN
    RAISE EXCEPTION 'Only an admin can archive (delete) an expense';
  END IF;

  -- Lock the expense; tenant-scoped, not already archived (idempotent).
  SELECT * INTO v_exp FROM public.expenses
   WHERE id = p_expense_id AND tenant_id = v_tenant AND deleted_at IS NULL
   FOR UPDATE;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'Expense not found or already archived';
  END IF;

  -- If money posted to the ledger, reverse every linked transaction with a
  -- compensating entry (append-only ledger) and retire its VAT rows so reports
  -- stop counting deleted spend (EXP-006).
  IF v_exp.status IN ('approved', 'paid') THEN
    FOR v_txn IN
      SELECT id FROM public.financial_transactions
       WHERE reference_type = 'expense' AND reference_id = p_expense_id
         AND deleted_at IS NULL
    LOOP
      PERFORM public.reverse_financial_transaction(
        v_txn.id, COALESCE(p_reason, 'Expense archived'));
    END LOOP;

    UPDATE public.vat_records
       SET deleted_at = now()
     WHERE record_id = p_expense_id
       AND record_type IN ('purchase', 'expense')
       AND deleted_at IS NULL;
  END IF;

  -- Cascade attachment soft-delete.
  UPDATE public.expense_attachments
     SET deleted_at = now()
   WHERE expense_id = p_expense_id AND deleted_at IS NULL;

  -- Archive the expense (audit trigger stamps updated_by; the authority trigger
  -- passes because this definer already admin-checked above).
  UPDATE public.expenses
     SET deleted_at = now(), status = 'voided'
   WHERE id = p_expense_id
   RETURNING * INTO v_exp;

  RETURN v_exp;
END;
$$;

REVOKE ALL ON FUNCTION public.archive_expense(uuid, text) FROM public;
GRANT EXECUTE ON FUNCTION public.archive_expense(uuid, text) TO authenticated;
```

### issues
- VERIFIED-CORRECT claims: deleteExpense (expensesService.ts:315-357) has ZERO callers (grep confirms only deleteExpenseAttachment matches); it already does full ledger reversal + VAT retirement + attachment cascade + status='voided'. handleBulkArchive (ExpensesList.tsx:386-389) is a raw multi-row .update({deleted_at}) with NO reversal/VAT/cascade = the real production hole. canBulkArchive=owner/admin (line 96) is UI-only. expenses_update RLS = has_role('accounts') (owner/admin/manager/accounts) and expenses_delete = has_role('admin') governs only hard DELETE (dead). status CHECK includes 'voided'. record_expense_disbursement/reverse_financial_transaction/has_role/is_platform_admin/get_current_tenant_id are all SECURITY DEFINER + search_path=public. vat_records has record_id/record_type/deleted_at. set_expenses_tenant_and_audit is BEFORE INSERT OR UPDATE — new BEFORE UPDATE trigger composes fine (both BEFORE; audit only stamps updated_at).
- WRONG RATIONALE (not a functional bug but the design's core justification is incorrect): the design says the archive UPDATE 'passes the authority trigger because this definer already admin-checked above.' That is NOT how it works. has_role() resolves via auth.uid() (profiles WHERE id=auth.uid()). SECURITY DEFINER changes the executing ROLE/privileges, NOT auth.uid()/the JWT. So the trigger INDEPENDENTLY re-runs has_role('admin') against the ORIGINAL CALLER. It passes only because the RPC caller is admin — the RPC's own admin check is therefore redundant with the trigger. Consequence: the trigger, not the RPC, is the true gate, and it fires for EVERY caller including non-RPC paths.
- BLAST RADIUS — maintenance/backfill regression: the new BEFORE UPDATE trigger raises on ANY NULL->non-NULL deleted_at transition unless has_role('admin') OR is_platform_admin(). set_tenant_and_audit_fields honors app.bypass_tenant_guard for backfills, but the new trigger does NOT. Any main-loop/service-role data script, deleted_at sweep (cf #311), or migration that archives an expense without an admin JWT context will now FAIL. This silently breaks documented operator backfill patterns.
- NON-ATOMIC BULK: handleBulkArchive rewritten as for(){await archiveExpense(id)} = one transaction per row. A mid-batch failure leaves some expenses archived (ledger reversed) and others not — no all-or-nothing. Design acknowledges per-row tally but this is a partial-state hazard on a financial mutation. A set-based archive_expenses(uuid[]) RPC would be atomic per call.
- OVER-ENGINEERING vs the actual bug: the only live defect is the bulk path skipping reversal. That is closeable with ZERO migration by routing handleBulkArchive through the existing, already-correct deleteExpense (looped) and tightening its callers — the single-row service path is already safe and unused. Introducing a permanent always-on BEFORE UPDATE trigger + new SECURITY DEFINER RPC + types regen to also tighten accounts->admin authority is a larger, higher-risk surface than the bug warrants. The authority change (accounts/manager lose soft-delete) is a real behavior change that should be an explicit product decision, not bundled into a bug fix.
- MIGRATION SAFETY: additive (CREATE OR REPLACE FUNCTION + CREATE TRIGGER), reversible (DROP TRIGGER/FUNCTION). CREATE TRIGGER takes a brief ACCESS EXCLUSIVE lock on expenses — negligible on a 1-row table. RESTRICTIVE tenant isolation preserved; RPC re-checks tenant+admin. The trigger function being SECURITY DEFINER is pointless (has_role/is_platform_admin are already definer) but harmless. Note: vat_records soft-delete inside the RPC writes to vat_records directly under definer — relies on the RPC's tenant scoping (record_id match) since RLS is bypassed; acceptable because record_id is the expense id already tenant-locked by the FOR UPDATE select.
- EXP-032 / sales-VAT path: UNAFFECTED. RPC only soft-deletes vat_records WHERE record_id=expense AND record_type IN ('purchase','expense'); never calls calculateVATForPeriod or touches sale rows. Banking: correctly does NOT reverse a real disbursement (called out as non-goal) — but note that means archiving a PAID expense reverses the GL accrual yet leaves the bank_transaction/cash debit standing, which can desync GL vs bank. The current deleteExpense has the same behavior, so not a regression, but worth an explicit guard/warning for status='paid'.
