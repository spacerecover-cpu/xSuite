# EXP-024-fx — VAT period queries key off created_at not economic date; tax_period unpopulated (EXP-032)
priority=High verdict=fix conf=0.82 needsMigration=True risk=medium

## summary
GROUNDING: The write-side of EXP-024 and EXP-032 already shipped. approveExpense (src/lib/expensesService.ts:424-454) posts the ledger at transaction_date = expense.expense_date (EXP-024 done) and calls createExpenseVATRecord (expensesService.ts:753-777) which writes record_type='purchase' with tax_period = expense_date.slice(0,7) (YYYY-MM) (EXP-032 write done). I verified live: vat_records.tax_period is text, nullable; all 21 active rows are record_type='sale' with tax_period='2026-06' already populated (a prior backfill); no DB trigger writes tax_period (only set_tenant_and_audit_fields). createVATRecordFromInvoice (vatService.ts:209-222) is the only sales writer and it OMITS tax_period — and it currently has ZERO callers anywhere in src (the live sale rows came from a since-removed/manual path). So the sales VAT write path is dormant in current code.

THE REMAINING BUG (EXP-032 read-side, still unfixed): vatService.ts aggregates/filters vat_records by created_at (INSERT time), not the economic period. calculateVATForPeriod (l.104-109), getVATRecordsByReturn (l.267-273), getQuarterlyVATSummary (l.279-295 via calculateVATForPeriod), and fetchVATRecords (l.45,52,56) all key off created_at. A Q1 expense approved Q2 reports in Q2. tax_period is written but never read.

THE FIX (safe slice): Make the period-scoped readers filter on tax_period instead of created_at. Because period inputs arrive as ISO dates (YYYY-MM-DD) but tax_period is YYYY-MM, derive a [startMonth, endMonth] inclusive month-string range and use .gte('tax_period', startMonth).lte('tax_period', endMonth). Add a defensive OR fallback so any row with NULL tax_period still buckets by created_at month (no silent data loss for legacy/edge rows). Stamp tax_period on createVATRecordFromInvoice so the dormant sales writer is correct if/when it is wired (defense in depth; no behavior change today since it has no callers). Backfill the 21 live sale rows' tax_period is already non-null, so the backfill is a no-op safety net keyed off the source invoice_date where resolvable — I scope the backfill conservatively (set tax_period = to_char(created_at,'YYYY-MM') only WHERE tax_period IS NULL) so the sales/invoice VAT path is provably untouched. This satisfies EXP-024-fx's intent (period queries key off the economic date; tax_period populated and consumed) while keeping the SHARED calculateVATForPeriod safe for the sales/invoice path.

DEFERRED (out of safe scope, stated explicitly): (1) storing a full economic DATE on vat_records rather than YYYY-MM — not needed; VAT returns are monthly/quarterly so month granularity is sufficient and matches the existing column. (2) Wiring the invoice approval path to actually call createVATRecordFromInvoice (that is a separate sales-VAT initiative, not an expense finding). (3) fetchVATRecords' dateFrom/dateTo on the VATAuditPage uses created_at — I switch its period filter to tax_period too for consistency, but its UI label is 'records' not 'period', so this is low-risk; if reviewers prefer minimal blast radius we can leave fetchVATRecords on created_at and only change the three period-math readers.

## blastRadius
SHARED SURFACE: calculateVATForPeriod is consumed by both expenses (input/purchase VAT) and the sales/invoice side via VATReturnModal.tsx:61 and getVATStats/createVATReturnFromPeriod/getQuarterlyVATSummary. Changing it from created_at to tax_period could in principle shift where SALE rows land. (b) The .or() keeps a created_at fallback for any NULL-tax_period row, so no row is ever silently dropped from a period total — the worst case for an un-backfilled row is identical to today's behavior. (c) The backfill runs NULL-only, so it cannot rewrite a sale row that already has a period. (d) createVATRecordFromInvoice has ZERO callers in current code, so adding the optional taxPeriod param changes no live behavior. Net: invoices/banking/reports outside vat_records are not touched (no other module reads vat_records except VATAuditPage display and VATReturnModal which both go through these functions). The one behavioral change a tenant could observe: a purchase expense whose expense_date and approval date straddle a month/quarter boundary now correctly reports in the expense-date period — which is the intended fix. PostgREST .or() with embedded interpolated month strings: months are derived by slice(0,7) from already-sanitized ISO dates (not user free-text), so no injection vector; still, keep the values constrained to /^\\d{4}-\\d{2}/ defensively in the helper.

### migrationSafety
The migration is data-only, additive, idempotent, and NULL-scoped (UPDATE ... SET tax_period=to_char(created_at,'YYYY-MM') WHERE tax_period IS NULL AND deleted_at IS NULL) — no DDL, so no types regen needed, and re-running is a no-op once tax_period is populated. Lock risk is negligible (current vat_records is ~21 rows). REVERSIBILITY: there is no down-path and the backfill is lossy in principle — it overwrites NULLs with a created_at-derived month, so if a row's true economic month differs from its created_at month, the backfill bakes in the WRONG period and you cannot tell which rows were backfilled vs authored. For the 21 live rows this is moot (all already non-null, so untouched). MISREPRESENTED SAFEGUARD: the SQL comment claims tenant isolation makes this safe, but apply_migration runs as the migration/owner role which BYPASSES RLS — this UPDATE hits EVERY tenant's NULL rows platform-wide, not the current tenant. That is acceptable for an idempotent NULL-only backfill but the design's stated mitigation is wrong and should not be relied on. tenant_id is NOT NULL on the table (verified), so no rows are orphaned.
