# EXP-010 — Zero audit trail for expense lifecycle; actor fields client-spoofable
priority=High verdict=fix conf=0.82 needsMigration=True risk=medium

## summary
After 310-317 the expenses table has only set_expenses_tenant_and_audit and no actor stamping so created_by is client supplied and spoofable and there is no log_audit_trail or log_case_history in the expense files. See files for the full design.

## files
### create `FINDINGS grounding`
Confirmed live: expenses carries only set_expenses_tenant_and_audit running set_tenant_and_audit_fields which stamps tenant_id and timestamps, never an actor column. The v1.2.0 set_audit_actor_fields was never attached. created_by is taken from the client at ExpensesList line 208, approved_by and rejected_by from profile id at lines 231 and 243, so a direct insert can spoof the actor. Grep shows zero log_audit_trail or log_case_history calls in the expense files. log_audit_trail is SECURITY DEFINER and stamps performed_by and tenant server-side; the wrapper logAuditTrail and log_case_history already exist.

### create `MIGRATION expense_actor_stamping (LAYER A, applied by MAIN LOOP)`
New SECURITY DEFINER trigger fn set_expense_actor_fields before insert and update on expenses, ordered after set_expenses_tenant_and_audit so tenant_id resolves first. INSERT forces created_by to auth uid, keeping a supplied value only when no auth context exists for system or import jobs. UPDATE sets updated_by to auth uid, pins created_by to OLD value so authorship is immutable, and stamps submitted_by approved_by or rejected_by to auth uid on entry into that status. Forcing rather than COALESCE makes the spoof impossible. Exact SQL in the migrationSql field. No DROP or destructive ops.

### edit `src/lib/expensesService.ts (LAYER B)`
Import logAuditTrail from auditTrailService and add a helper calling the log_case_history rpc. createExpense after insert at line 216 logs created with status amount category_id. updateExpense widens the pre image select at line 233 to also fetch amount category_id status and logs updated with changed_fields equal to the changed keys plus old pre image before the return at line 312. submitExpense logs submitted. approveExpense logs approved before the createFinancialTransaction call and when case_id is set also calls log_case_history expense_approved. rejectExpense logs rejected with rejection_reason. markExpenseAsPaid logs paid and case-linked also log_case_history expense_paid. deleteExpense after the final update at line 351 logs deleted with new status voided. recordExpenseDisbursement logs disbursed.

### edit `src/lib/expensesService.ts (error policy)`
The create update and approve audit calls throw for fail fast parity with the ledger so books and trail stay consistent. The delete markPaid and disbursement audit calls are wrapped in try with logger error so a logging failure never blocks the soft-delete or the payment. The existing created_by and approvedBy params are kept for signature stability but are now advisory because the DB trigger forces the real value.

### edit `src/types/database.types.ts`
Regenerate via the supabase MCP after the migration. Trigger only change so likely zero table row type diff, but regenerate and commit per migration discipline or confirm a zero diff.

## migrationSql
```sql
CREATE OR REPLACE FUNCTION public.set_expense_actor_fields()
  RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER SET search_path TO 'public'
AS $BODY$
DECLARE uid uuid := auth.uid();
BEGIN
  IF TG_OP = 'INSERT' THEN
    IF uid IS NOT NULL THEN NEW.created_by := uid; END IF;
  ELSIF TG_OP = 'UPDATE' THEN
    NEW.updated_by := COALESCE(uid, NEW.updated_by);
    NEW.created_by := OLD.created_by;
    IF NEW.status='pending'  AND OLD.status IS DISTINCT FROM 'pending'  AND uid IS NOT NULL THEN NEW.submitted_by := uid; END IF;
    IF NEW.status='approved' AND OLD.status IS DISTINCT FROM 'approved' AND uid IS NOT NULL THEN NEW.approved_by  := uid; END IF;
    IF NEW.status='rejected' AND OLD.status IS DISTINCT FROM 'rejected' AND uid IS NOT NULL THEN NEW.rejected_by  := uid; END IF;
  END IF;
  RETURN NEW;
END;
$BODY$;
DROP TRIGGER IF EXISTS zset_expenses_actor_fields ON public.expenses;
CREATE TRIGGER zset_expenses_actor_fields BEFORE INSERT OR UPDATE ON public.expenses
  FOR EACH ROW EXECUTE FUNCTION public.set_expense_actor_fields();
```

## VERDICT
fix (conf 0.82)

### issues
- log_audit_trail has TWO overloads on the live DB (6-arg and 8-arg with p_ip_address/p_user_agent). Calling via supabase-js named params can trigger PostgREST PGRST203 'could not choose best candidate function' ambiguity. Design assumes a single clean callable.
- The existing logAuditTrail wrapper (src/lib/auditTrailService.ts) takes only 5 args and does NOT pass p_changed_fields. The design's updateExpense 'logs updated with changed_fields' cannot use the wrapper as-is; the wrapper must be extended or a new call added.
- approveExpense error policy is backwards: design makes the audit call throw 'for parity with the ledger', but if the audit call is placed before createFinancialTransaction and throws, the expense is already committed as approved (separate non-transactional calls) with NO ledger row — the exact inconsistency the ledger fail-fast prevents. Approve audit logging must be best-effort, not throw.
- INSERT only forces created_by when auth.uid() IS NOT NULL; a NULL-auth path (service-role/edge fn) can still supply/spoof created_by, and legitimate import jobs (known issue, mem 8738) running under a real auth context cannot attribute to the real submitter. Interaction unaddressed.
- submitExpense already sets submitted_by from a client param and updateExpense's resubmit path sets status='pending'; the new trigger will now overwrite submitted_by with auth.uid() on every pending transition — a behavior change (mostly desirable) not called out in the design.
- Trigger-ordering rationale is incorrect: actor fn does not depend on tenant_id resolution; ordering is irrelevant (harmless but indicates incomplete reasoning).

### mustFixBeforeShip
- Resolve the log_audit_trail overload ambiguity: call the specific signature explicitly (provide all 6 named args incl. p_changed_fields, or disambiguate) and confirm no PGRST203; verify the typed RPC signature in database.types.ts after regen.
- Extend logAuditTrail wrapper (or add a call) to actually pass p_changed_fields, since updateExpense's design depends on it.
- Make the approveExpense audit/log_case_history calls best-effort (try/catch + logger), NOT fail-fast — throwing there leaves an approved expense with no ledger row. Only create/update may throw without breaking ledger consistency.
- Document/handle the submitted_by overwrite on the updateExpense status->pending resubmit path so EXP-016 resubmit behavior is intentional.

### migrationSafety
Migration is additive and safe on the live DB. It only CREATEs set_expense_actor_fields (SECURITY DEFINER, SET search_path=public, correct) and a BEFORE INSERT/UPDATE trigger; no DROP of data, no column/table drops. The trigger writes ONLY actor columns (created_by/updated_by/submitted_by/approved_by/rejected_by) — verified all five exist and are nullable on expenses — never amount/currency/status/tenant_id/updated_at, so EXP-006 void, EXP-019 optimistic lock (uses updated_at, untouched), FX snapshots, invoices, sales VAT, banking and reports are all unaffected. Reversible: DROP TRIGGER + DROP FUNCTION. Lock: CREATE TRIGGER takes a momentary ACCESS EXCLUSIVE lock on expenses, trivial at current volume (1 row). RESTRICTIVE tenant isolation intact; no cross-tenant path since log_audit_trail/log_case_history stamp tenant via get_current_tenant_id() server-side (both verified SECURITY DEFINER). Not duplicate work: confirmed live that only set_expenses_tenant_and_audit exists and set_tenant_and_audit_fields never sets any actor column; the v1.2.0 set_audit_actor_fields was never attached to expenses. One non-fatal nit: the 'order after tenant trigger so tenant_id resolves first' rationale is wrong — the actor fn reads only auth.uid()/status, not tenant_id, so the zset naming is harmless but the justification is incorrect.
