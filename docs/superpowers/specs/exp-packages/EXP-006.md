## summary
EXP-006's original four sub-gaps have ALREADY been remediated in code I verified on this branch (#310–#317): updateExpense throws on approved/paid/voided (expensesService.ts:231-241); deleteExpense reverses every linked ledger row via reverse_financial_transaction + soft-deletes the VAT row + sets status='voided' (252-357); approveExpense posts the ledger at expense_date not approval date (424-437) behind a pending-only state guard + SoD; purchase-VAT with tax_period is written (439-454, createExpenseVATRecord 753-777). Live DB confirms the supporting invariants exist: uq_financial_transactions_expense_ref (partial unique on reference_type='expense'), expenses_status_chk including 'voided', and reverse_financial_transaction (SECURITY DEFINER, compensating NEGATIVE-amount row with reference_type='reversal', transaction_type copied from the original). financialReportsService P&L/cash-flow/revenue-by-case all read the expenses table and already filter deleted_at (lines 105/252/416/497/584/641) — so the audit's "pick one source of truth = keep reports on the expenses table" is effectively in force, and getTransactionStats/getCashFlowSummary correctly net the negative reversal because it carries transaction_type='expense'.

What EXP-006 STILL lacks is the part its title names explicitly: "no single source of truth + NO RECONCILIATION TEST." There is no function or test that asserts the expenses table and the financial_transactions ledger actually tie out, so the invariant can silently rot on the next change. There is also one real residual divergence: reverse_financial_transaction posts the compensating row at now(), while the original posted at expense_date — when a delete/void crosses a month boundary the per-PERIOD ledger nets wrong (original month over-counts, deletion month under-counts) even though the all-time net is zero; and deleteExpense's reversal/VAT/expense writes are several non-atomic client round-trips that desync on partial failure.

This package delivers the SAFE remaining slice: (1) a SECURITY DEFINER reconciliation function reconcile_expense_ledger() returning per-period and total expense-table-vs-ledger deltas (the executable "single source of truth" contract); (2) a thin read-only service wrapper getExpenseLedgerReconciliation() in expensesService.ts; (3) an atomic void_expense(p_expense_id, p_reason) SECURITY DEFINER RPC that does the reverse-ledger + retire-VAT + soft-delete + status='voided' in one transaction and posts the reversal at the ORIGINAL row's transaction_date (fixing the period residual), with deleteExpense rewired to call it; (4) tests asserting the reconciliation math (including reversal-aware netting) and the void atomicity wrapper. It does NOT touch the shared sales/invoice VAT path (EXP-032 deferred) and does not migrate any reader to the ledger.

```sql
-- EXP-006: reconciliation function + atomic void RPC for the expense↔ledger relationship.
-- Both SECURITY DEFINER with SET search_path=public; authorize via has_role/tenant fns
-- because RLS is bypassed under definer. No table DDL — purely additive functions.

-- 1) Read-only reconciliation: per-period and total delta between the expenses table
--    (the chosen source of truth) and the financial_transactions expense ledger.
--    Ledger side sums transaction_type='expense' AND deleted_at IS NULL, which already
--    INCLUDES the negative reference_type='reversal' compensating rows, so a voided
--    expense (excluded on the expenses side) nets to 0 on the ledger side too.
CREATE OR REPLACE FUNCTION public.reconcile_expense_ledger(
  p_date_from date DEFAULT NULL,
  p_date_to   date DEFAULT NULL
)
RETURNS TABLE (
  period            text,
  expenses_base     numeric,
  ledger_base       numeric,
  delta_base        numeric
)
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_tenant uuid;
BEGIN
  v_tenant := get_current_tenant_id();
  IF v_tenant IS NULL THEN
    RAISE EXCEPTION 'reconcile_expense_ledger: no tenant context'
      USING ERRCODE = 'insufficient_privilege';
  END IF;
  -- Reading financial data is staff-level; gate to non-viewer staff.
  IF NOT is_staff_user() THEN
    RAISE EXCEPTION 'reconcile_expense_ledger: insufficient role'
      USING ERRCODE = 'insufficient_privilege';
  END IF;

  RETURN QUERY
  WITH exp AS (
    SELECT to_char(e.expense_date, 'YYYY-MM') AS period,
           SUM(COALESCE(e.amount_base, e.amount)) AS base
    FROM expenses e
    WHERE e.tenant_id = v_tenant
      AND e.deleted_at IS NULL
      AND e.status IN ('approved','paid')
      AND (p_date_from IS NULL OR e.expense_date >= p_date_from)
      AND (p_date_to   IS NULL OR e.expense_date <= p_date_to)
    GROUP BY 1
  ),
  led AS (
    SELECT to_char(ft.transaction_date, 'YYYY-MM') AS period,
           SUM(COALESCE(ft.amount_base, ft.amount)) AS base
    FROM financial_transactions ft
    WHERE ft.tenant_id = v_tenant
      AND ft.deleted_at IS NULL
      AND ft.transaction_type = 'expense'
      AND (p_date_from IS NULL OR ft.transaction_date >= p_date_from)
      AND (p_date_to   IS NULL OR ft.transaction_date <= p_date_to)
    GROUP BY 1
  )
  SELECT COALESCE(exp.period, led.period) AS period,
         COALESCE(exp.base, 0)            AS expenses_base,
         COALESCE(led.base, 0)            AS ledger_base,
         COALESCE(exp.base, 0) - COALESCE(led.base, 0) AS delta_base
  FROM exp FULL OUTER JOIN led ON exp.period = led.period
  ORDER BY 1;
END;
$$;

REVOKE ALL ON FUNCTION public.reconcile_expense_ledger(date, date) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.reconcile_expense_ledger(date, date) TO authenticated;

-- 2) Atomic void: reverse every posted ledger row, retire the VAT row, soft-delete
--    attachments + the expense, and flip status to 'voided' in ONE transaction.
--    Fixes the deleteExpense non-atomic client round-trips and the period residual:
--    the reversal is posted at the ORIGINAL ledger row's transaction_date (the
--    expense_date), not now(), so per-period reconciliation nets to 0 in-period.
CREATE OR REPLACE FUNCTION public.void_expense(
  p_expense_id uuid,
  p_reason     text DEFAULT NULL
)
RETURNS expenses
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_tenant uuid;
  v_uid    uuid;
  v_exp    expenses%ROWTYPE;
  v_ft     financial_transactions%ROWTYPE;
  v_now    timestamptz := now();
BEGIN
  v_tenant := get_current_tenant_id();
  IF v_tenant IS NULL THEN
    RAISE EXCEPTION 'void_expense: no tenant context' USING ERRCODE = 'insufficient_privilege';
  END IF;
  v_uid := auth.uid();

  -- Removing financial records is the admin-gated authority (matches expenses_delete
  -- intent and EXP-018's recommendation to align soft-delete with the admin gate).
  IF NOT is_tenant_admin() THEN
    RAISE EXCEPTION 'void_expense: only an admin may void/delete an expense'
      USING ERRCODE = 'insufficient_privilege';
  END IF;

  SELECT * INTO v_exp FROM expenses
  WHERE id = p_expense_id AND tenant_id = v_tenant AND deleted_at IS NULL
  FOR UPDATE;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'void_expense: expense % not found', p_expense_id
      USING ERRCODE = 'foreign_key_violation';
  END IF;

  -- Reverse every posted (non-reversal) ledger row for this expense, dating the
  -- compensating entry to the ORIGINAL row's transaction_date for in-period netting.
  IF v_exp.status IN ('approved','paid') THEN
    FOR v_ft IN
      SELECT * FROM financial_transactions
      WHERE tenant_id = v_tenant
        AND reference_type = 'expense'
        AND reference_id = p_expense_id
        AND deleted_at IS NULL
        AND NOT EXISTS (
          SELECT 1 FROM financial_transactions r
          WHERE r.reference_type = 'reversal'
            AND r.reference_id = financial_transactions.id
            AND r.deleted_at IS NULL
        )
    LOOP
      INSERT INTO financial_transactions (
        tenant_id, transaction_type, amount, currency, transaction_date,
        description, category_id, reference_type, reference_id, bank_account_id,
        exchange_rate, rate_source, amount_base, status, notes, created_by
      ) VALUES (
        v_tenant, v_ft.transaction_type, -v_ft.amount, v_ft.currency, v_ft.transaction_date,
        'Reversal of ' || COALESCE(NULLIF(v_ft.description,''), v_ft.id::text)
          || COALESCE(' — ' || NULLIF(p_reason,''), ''),
        v_ft.category_id, 'reversal', v_ft.id, v_ft.bank_account_id,
        v_ft.exchange_rate, v_ft.rate_source,
        -v_ft.amount_base, 'posted', p_reason, v_uid
      );

      IF v_ft.bank_account_id IS NOT NULL THEN
        UPDATE bank_accounts SET current_balance =
          COALESCE(current_balance, 0)
          - CASE WHEN v_ft.transaction_type = 'income' THEN v_ft.amount ELSE -v_ft.amount END
        WHERE id = v_ft.bank_account_id AND tenant_id = v_tenant;
      END IF;
    END LOOP;

    -- Retire the input-VAT row so VAT period totals stop counting voided spend.
    UPDATE vat_records SET deleted_at = v_now
    WHERE tenant_id = v_tenant
      AND record_id = p_expense_id
      AND record_type IN ('purchase','expense')
      AND deleted_at IS NULL;
  END IF;

  UPDATE expense_attachments SET deleted_at = v_now
  WHERE expense_id = p_expense_id AND tenant_id = v_tenant AND deleted_at IS NULL;

  UPDATE expenses SET deleted_at = v_now, status = 'voided'
  WHERE id = p_expense_id AND tenant_id = v_tenant
  RETURNING * INTO v_exp;

  RETURN v_exp;
END;
$$;

REVOKE ALL ON FUNCTION public.void_expense(uuid, text) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.void_expense(uuid, text) TO authenticated;
```

### issues
- CRITICAL money corruption: void_expense does NOT reverse a PAID expense's cash side. record_expense_disbursement records the disbursement as a bank_transactions row + payment_disbursements row and debits bank_accounts.current_balance/current_balance_base DIRECTLY; the financial_transactions accrual row posted at approval has bank_account_id = NULL. void_expense's bank-balance reversal is gated on `IF v_ft.bank_account_id IS NOT NULL`, so for a paid expense it credits nothing back, and it never touches the bank_transactions row, the completed payment_disbursements row, or current_balance_base. Voiding a paid expense leaves the bank permanently over-debited and an orphan 'completed' disbursement. The design's blastRadius claim that void_expense 'reverses the original debit ... same effect the existing reverse RPC has' is FALSE for paid expenses.
- HIGH: void_expense hand-rolls a negative-amount INSERT into financial_transactions instead of calling reverse_financial_transaction, purely to back-date the reversal to the original transaction_date. This (a) violates the stated hard constraint 'reverse only via reverse_financial_transaction', (b) duplicates the exact compensating-row + bank-balance logic that will drift from the canonical RPC, and (c) back-dates a financial mutation into a possibly-closed prior accounting/VAT period. The shipped reverse_financial_transaction deliberately posts at now() for exactly this reason; the design treats that as a bug without weighing closed-period correctness.
- MEDIUM: createFinancialTransaction (approveExpense) posts the accrual ledger row with NO bank_account_id, so the design's whole premise that voiding nets the ledger AND the bank balance is only ever true for the (non-existent) case of a directly-bank-linked expense ledger row. The cash side lives entirely outside financial_transactions, which the reconcile_expense_ledger function also never checks — so reconciliation will report delta=0 while the bank is actually wrong.
- MEDIUM: vat_records has no unique constraint on record_id and record_type='purchase' is the SHARED sales/purchase VAT type; void_expense deletes by (record_id, record_type IN ('purchase','expense')). Safe today only because expense record_ids are expense PKs, but it is an unguarded shared-namespace delete on the VAT table that EXP-032 is explicitly deferring — fragile coupling.
- LOW: deleteExpense changes signature from void to returning the voided expenses row; design flags but does not verify the ExpensesList.tsx single-delete and bulk-archive callers tolerate it. Also void_expense's admin-only gate (is_tenant_admin) is STRICTER than the current deleteExpense path (no role gate) and stricter than record_expense_disbursement's has_role('accounts') — a silent permission tightening that may break existing manager/accounts delete flows.
- NOTE (safe part): the reconcile_expense_ledger function and getExpenseLedgerReconciliation wrapper are genuinely net-new, additive, read-only, correctly authorized inside the definer (is_staff_user verified live), and tenant-scoped. The pure-function reconciliation test is sound. This slice could ship on its own.
