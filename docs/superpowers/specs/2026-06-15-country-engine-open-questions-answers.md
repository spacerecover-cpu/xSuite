## Q1 — Pin the reference-data sources + holiday provider + refresh ownership for the Country Engine

**Recommendation: Pin a single open-data stack, all ingested into Postgres at build/seed time — no live third-party API in the request path.** Use Unicode CLDR (`cldr-core` + `cldr-numbers-full` + `cldr-dates-full`) as the keystone for locale / date / time / number / currency-display / week-start / weekend definitions; `iso-3166` / ISO 4217 codes (free, also re-published via CLDR) for countries, subdivisions and currency codes; `libphonenumber-js` for phone format/validation; Google libaddressinput (via `i18n-postal-address`) for per-country address layout and postal regex; IANA tz resolved at RUNTIME via `Intl`/`Temporal` + Postgres `pg_timezone_names` (never compute DST yourself); and **`date-holidays`** for public holidays — the only free provider with offline Umm al-Qura Eid computation for all 6 GCC states (1970–2080). Generate 2–3 forward years of concrete holidays into a new `geo_public_holidays` table with `is_estimated` / `source` / `confirmed_at` columns, and give GCC-DEEP tenants an admin "confirm/override holiday" surface for crescent-sighting Eid corrections.

**Rationale.** The engine's promise is "maintained external datasets, no country-specific code," which decomposes into two different problems. (a) Rules that change on a schedule you don't control (DST, currency decimals, locale formats) must be resolved at runtime by libraries that ship their own updates — storing them as your own data guarantees staleness. (b) Facts stable enough to snapshot (country codes, address layout, phone metadata, this-year's holidays) get ingested into `geo_*` at seed time so the request path has zero external dependency and stays inside your single data-residency region. CLDR is the keystone because it is the one dataset that simultaneously answers currency-display, number format, date/time pattern, week-start AND weekend-day — collapsing five of eight requirements into one Unicode-3.0 dependency, and it is the channel ISO 4217 currency-decimal data flows through, so you never touch the paywalled ISO PDFs. The holiday provider is where most teams get this wrong: Nager.Date is free and MIT but DOCUMENTS that it omits Eid al-Fitr/Eid al-Adha (moon-sighting dependent) — precisely the holidays that matter in your three deepest launch markets. `date-holidays` computes them via Umm al-Qura offline through 2080. What most people miss: holidays are the ONLY part of this stack that needs an ongoing human owner — everything else is a dependabot bump. Eid dates are officially corrected on crescent-sighting days before the event, so you need an `is_estimated` flag and a tenant-facing override, not a belief that any dataset is authoritative ahead of the sighting.

## Q2 — GCC statutory compliance: BUILD vs BUY + budget + named-owner model

**Recommendation: SPLIT along public RULES vs regulated TRANSMISSION.** BUILD (hand-author in-config) all statutory rules for KSA/UAE/Oman, because they are public and finite: VAT (KSA 15% / SAR 40M monthly-vs-quarterly threshold / 15-digit TRN; UAE 5% / AED 150M threshold / VAT201 / 15-digit TRN; Oman 5%); payroll (KSA GOSI, Oman SPF, SANED, EOSB); WPS files from published layouts — into effective-dated tables, with a NAMED external GCC tax/payroll advisor on retainer signing off every rule change. BUY a certified ASP ONLY for the regulated real-time e-invoicing transmission/clearance layer, and **defer it behind a named live-wave customer**: design the invoice model to PINT-AE / ZATCA-UBL / Peppol-BIS shape now with a clean adapter seam, and integrate one ASP per country only when a tenant crosses a live mandate. Do NOT buy an enterprise Sovos/Fonoa/Pagero contract now, and do NOT build the ZATCA crypto pipeline now. This makes Phase-3 RULES a matter of WEEKS; transmission is quarters of calendar but weeks of your effort, deferred to a paying customer.

**Rationale.** Three findings drive it. (a) Rules are public and hand-authorable for 3 countries: ZATCA, FTA, GOSI, Oman SPF, MOHRE all publish rates, thresholds, formats and EOSB/WPS layouts — a finite citable table the effective-dated tables and payroll registry already target. (b) ZATCA Phase 2 does NOT require a listed provider: a taxpayer meeting the requirements is compliant even if its provider is not in the directory, so BUY is not legally forced; the directory exists because the crypto pipeline (CSR, OTP, CCSID, compliance test, production CSID, SHA-256, UBL signing, PIH chaining, B2B clearance vs B2C 24h reporting) is hard and a way to ship a wrong cryptographic stamp to a tax authority — the one place to buy, but only when a tenant hits a live wave. (c) Timing allows deferral: KSA Wave 24 (over SAR 375k, 30-Jun-2026); UAE voluntary pilot Jul-2026, mandatory 1-Jan-2027 (over AED 50M) then 1-Jul-2027, ASP deadline extended to 30-Oct-2026; Oman Fawtara Phase 1 Aug-2026 is the 100–150 largest only, all VAT-registered not until Aug-2027. A SaaS onboarding its first GCC tenant in 2026 is below these thresholds, so transmission is built behind a customer gate without breaking any law. The quarters-long work (procurement, accreditation, certification) defers; the weeks-long work (rules + adapter seam + named reviewer) ships in Phase 3.

## Q3 — Customer portal + transactional email language: per-tenant vs per-recipient

**Recommendation: Go PER-RECIPIENT.** Store an explicit language preference on the customer record and resolve every portal render and outbound email through a single fallback chain — NOT a per-tenant global. This is how Stripe, Shopify, and Intercom do it, and the GCC mixed-audience case (English corporate client + Arabic individual under one tenant) is unsolvable with a tenant-level setting. Minimal additive migration: `ALTER TABLE customers_enhanced ADD COLUMN preferred_language text NULL` (BCP-47 vocabulary from `geo_countries.language_code`, e.g. 'en','ar'; NULL = fall back; plain text validated against the supported set, NOT an enum). Add `preferred_language` to the `PortalCustomer`/`PortalSession` shape and have `authenticate_portal_customer` return it. Resolution chain (implement once, reuse for portal UI + email + PDF): `customers_enhanced.preferred_language` → portal session's last explicit in-app switch → tenant default language → `geo_countries.language_code` (via customer's `country_id`, already FK'd) → 'en'. For email, keep rendering in the caller (the `send-document-email` edge function stays a dumb relay) and pass only an optional `locale` field through to it and into `log_case_communication` for forensic auditability; defer server-side per-(template, locale) rendering until the `feat/pdf-template-engine` work lands a real multi-language template catalog.

**Rationale.** The mixed-audience case is dispositive: a single GCC tenant emailing an English corporate client and an Arabic individual cannot be served by a per-tenant language setting — by definition the two recipients need different languages from the same tenant. Every leading global SaaS resolves this at the recipient level: Stripe puts `preferred_locales` (an ordered array) on the Customer object and localizes invoice/receipt/credit-note emails and PDFs; Shopify drives notification language off `Customer.locale`; Intercom uses a per-contact Language Override with an explicit priority chain. The codebase is ready: `customers_enhanced` already FKs `country_id` to `geo_countries`, which carries `language_code` — so the country-language fallback rung needs no new join, and the column is purely additive (CLAUDE.md mandates additive-only). What most people miss: the expensive part is NOT the schema (the column is a 1-line additive migration) — it's that every transactional template and every PDF document builder must exist in each language with a missing-translation fallback that is silent-in-prod / loud-in-dev. With only en/ar UI today but a 13-language PDF catalog, the column lets you LIGHT UP per-recipient language incrementally per template without ever re-migrating. Ship the column + resolver + audit-the-locale now; localize templates as they get translated.

## Q4 — Default LEGAL ENTITY for a case once a tenant has more than one entity (Phase 4 multi-entity)

**Recommendation: Adopt the proposed rule exactly.** (1) Exactly 1 entity: auto-assign the tenant's primary entity silently, no picker. (2) More than 1 entity: pre-select the user's/branch's home entity as the default BUT require an explicit, visible confirmation at case creation (the entity field is shown, defaulted, and must be deliberately submitted — never silently committed). (3) Lock the entity on the case the moment the first financial document (first numbered quote OR invoice, whichever issues first) is generated against it; before that it stays editable by manager+, after that any change requires a void/reissue path. Make `legal_entity_id` a first-class mandatory NOT NULL FK on `cases` once the tenant has more than one entity. This mirrors NetSuite OneWorld, Odoo, Xero, and Zoho Books: none let a posted/numbered tax document silently float to the wrong entity, and all pin the entity to a stable parent plus the number series.

**Rationale.** Ship the rule as written — it is exactly where the mature multi-entity ERPs converge, and the failure mode is unrecoverable by edit. A tax document issued under the wrong legal entity means the wrong VAT/GSTIN, wrong currency, and wrong invoice number series — a statutory misstatement in KSA/UAE/Oman where e-invoicing is enforced. NetSuite OneWorld DERIVES subsidiary from the entity and treats it as immutable once a transaction posts (it drives the GL, period and tax). Xero takes the hardest line: separate legal entities cannot share a ledger — each is a separate Organisation with its own chart of accounts and number series. Odoo auto-stamps the active company on new transactional records but DROPPED its silent default-company-on-partner behavior in v13 precisely because silent defaults created wrong-company records, and hard-blocks cross-company access. Zoho Books binds a unique transaction series to each branch/GSTIN and auto-applies it from the selected branch. The common thread: the entity is always either derived from a stable parent or explicitly chosen, and pinned hard once money/tax documents exist. xSuite is case-first, not transaction-first, so a case is created before any document — you cannot derive the entity from a posted transaction the way NetSuite does. That is exactly why you need explicit confirmation at case creation, then lock-on-first-document to recover NetSuite's immutability guarantee. What most people miss: the dangerous default is not no-default, it is a default that LOOKS confirmed but was not. Pre-select the home entity (good UX), force a deliberate submit, and make the entity visually prominent (name + tax-regime + currency badge) so a UAE-branch user creating a case that should bill from the KSA entity sees the mismatch before issuing a ZATCA invoice. The lock boundary must be first financial document issued, not case closed, because tax regime, currency and number series are consumed the instant a quote/invoice is numbered.

| Option | Pros | Cons | Cost / License |
|---|---|---|---|
| Silent home-entity default, editable until close (rejected) | Fastest UX, zero clicks; fewest schema/UI changes | The exact anti-pattern. A silent default never explicitly confirmed is how wrong-entity tax documents get issued; Odoo specifically REMOVED its silent partner-company default for this reason; in GCC e-invoicing a wrong-entity invoice is a statutory misstatement; editable-until-close lets the entity move after a numbered invoice exists, orphaning a real tax document | Lowest build cost, highest compliance/legal risk; net negative |
| **RECOMMENDED:** pre-select home entity + REQUIRE explicit confirmation + lock on first financial document | Matches the convergent ERP pattern (NetSuite immutability-on-post, Zoho series-per-entity, Odoo explicit company stamp); keeps fast UX (1 entity invisible; >1 pre-filled) while making the >1 case a deliberate visible choice; lock boundary aligns with when tax/currency/number-series are consumed; entity choice becomes a timestamped audit event | Requires a mandatory entity field in CreateCaseWizard, a NOT NULL FK, a lock derived from first quote/invoice, and a guarded re-point path (void+reissue); adds one confirmation step for multi-entity tenants | No new license. Medium build, ~4–7 dev-days incl. tests |
| Forced selection with NO default (NetSuite-strict, no pre-fill) | Zero risk of an accidental wrong default; user must consciously choose every time | Worse UX for the common single-home-branch user who picks the same entity every time, training users to click through blindly and defeating the safety goal; loses the helpful default signal | Similar build cost to recommended but worse adoption; not warranted |
| Separate ledger per entity, explicit org-switch (Xero model) | Strongest isolation; impossible to cross entities on a single document; cleanest statutory separation | Architecturally wrong for xSuite: breaks the single-tenant case workbench, cross-entity reporting, shared customers/devices, and the unified custody chain; Xero gets away with it because it is single-company-per-org by design; xSuite is explicitly multi-entity-within-tenant (Phase 4) | Very high re-architecture cost; contradicts locked Phase 4 direction. Reject |

**Sources**
- NetSuite OneWorld: a transaction is assigned the same subsidiary as its entity; for transactions without an entity you must manually select a subsidiary; each transaction is tied to one subsidiary — https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/section_N550454.html
- NetSuite: you must assign a subsidiary to each entity record before entering transactions; a transaction inherits its subsidiary from its entity, which governs form fields, address format and tax fields — https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/section_N276477.html
- NetSuite Multi-Subsidiary Customer: the transaction defaults to the customer's primary subsidiary and can be overridden to any assigned secondary subsidiary — https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/section_161895998055.html
- Odoo: the current company is auto-stamped on new transactional records; products and contacts default blank (shared); if a related document exists the new record inherits its company; the default-company-on-partner proposal was removed in v13 — https://www.odoo.com/documentation/18.0/applications/general/multi_company.html
- Xero: each legal entity must be a separate Organisation with its own subscription, chart of accounts and number series; you cannot run multiple legal entities in one ledger; switch organisations explicitly — https://central.xero.com/s/article/Add-another-organisation-to-Xero-AU
- Zoho Books Branches: each GSTIN/branch can have a unique invoice number series applied automatically based on the branch selected; each GSTIN must have a unique transaction series — https://www.zoho.com/in/books/help/branches/basic-functions.html

**Cost estimate.** No new license or vendor cost; an internal design rule. Build ~4–7 engineer-days: (a) schema — add `legal_entity_id uuid NOT NULL` FK to `cases` (default-backfilled to the tenant primary entity), scope `number_sequences` per entity, derive an entity lock from the existence of any quote/invoice; (b) CreateCaseWizard — entity step that auto-fills and hides when 1 entity, pre-selects home entity and requires explicit submit when >1, with a tax-regime/currency/number-series badge; (c) an audited manager+ re-point flow for the pre-document window and a hard block post-document; (d) `case_job_history` event on entity assignment/change. Ongoing cost negligible.

**What it unblocks.** Unblocks Phase 4 multi-entity case creation and the Country Engine's tax/currency/number-series binding: once the entity is a confirmed, locked case attribute, quote/invoice generation can deterministically resolve tax regime (ZATCA vs UAE VAT vs Oman), currency, and the correct per-entity invoice number series without ambiguity. Also unblocks correct chain-of-custody and reporting attribution per legal entity.

**Confidence:** High.

**Still your call:** (1) Lock-boundary precision: confirm first financial document means first NUMBERED quote OR first invoice. If quotes are non-statutory drafts that can be discarded freely, lock only on first invoice (or first quote-acceptance); if quotes carry the entity's branding/number series, lock on first quote. (2) Home-entity source: decide whether the pre-selected default comes from the user's profile (assigned branch/entity) or the active branch context; for multi-branch staff, profile-level home entity is usually safer. (3) Re-point authority: confirm which role can re-point before lock (recommend manager+, audited) and that post-lock re-pointing is fully disallowed (void+reissue only), consistent with the append-only custody/audit posture.

## Q7 — Does collapse-now satisfy "depth delivered incrementally"?

**Recommendation: YES — collapse-now satisfies locked decision #1 ("depth delivered incrementally"); the only thing missing is the owner's explicit sign-off that "dormant foundation now, activate on demand" matches intent.**

**Rationale.** The proposed approach is to ship the full 6-level entity hierarchy SCHEMA in Phase 1, but dormant: every existing tenant collapses to exactly one `legal_entity` and one business-unit, the new hierarchy columns are nullable, and live sub-unit RLS isolation is OFF. Depth is then delivered incrementally as Phase 4 work-streams — live RLS isolation, multi-entity operations, and multi-region — each gated behind a named customer that actually needs it. This is "additive + phased + gated," which IS the definition of incremental depth: the structural foundation lands once (no future re-migration, no breaking change to the locked schema discipline), tenants experience zero behavioral change on day one, and each layer of real depth activates only when a paying customer pulls it. That is materially safer than either extreme — building all six levels live now (premature complexity, unused RLS surface area, runway burn before any customer needs multi-entity) or deferring the schema entirely (guaranteed breaking migration later, which the additive-only / append-only discipline in CLAUDE.md forbids). Shipping the schema dormant means the expensive, hard-to-reverse part (the relational shape) is paid once and correctly; the cheap, reversible part (flipping isolation flags, lighting up multi-entity ops) happens per named customer. So the approach honors locked decision #1 directly: depth is real, it is delivered, and it is delivered incrementally rather than all-at-once. The residual is not technical — it is intent confirmation: the owner must explicitly agree that "dormant-foundation-now / activate-on-demand" is what "depth delivered incrementally" was meant to authorize, rather than a slower level-by-level live build.

| Option | Pros | Cons | Cost / License |
|---|---|---|---|
| **RECOMMENDED:** ship full 6-level schema dormant in Phase 1; activate depth per Phase-4 work-stream behind a named customer | Pays the hard-to-reverse relational shape once; additive + nullable so zero breaking migration later; zero day-one behavior change for existing tenants; honors locked decision #1 (additive + phased + gated = incremental depth); each activation is demand-driven, no runway burned on unused depth | Requires discipline to keep dormant columns nullable and isolation OFF until activated; "schema exists but inactive" must be clearly documented so no one assumes it's live; needs the owner's explicit intent sign-off | No new license. Schema-only Phase-1 cost (small migration); activation cost paid per work-stream when a customer triggers it |
| Build all six levels LIVE now (full multi-entity + RLS isolation in Phase 1) | Maximum capability immediately; no later activation work | Premature complexity and unused RLS attack surface; runway burned before any customer needs multi-entity; contradicts "incrementally" | High build cost now for no near-term customer |
| Defer the schema entirely; add levels when first needed | Nothing built until needed | Guaranteed breaking migration later (forbidden by additive-only / append-only discipline); re-points existing data; highest long-run risk | Low now, high later (re-migration + data fix) |

**Cost estimate.** No new license. Phase-1 cost is the dormant-schema migration only (small, additive, nullable columns, single `legal_entity` + business-unit backfill per existing tenant). Each depth activation (live RLS isolation, multi-entity ops, multi-region) is a separate Phase-4 work-stream costed when its named customer lands.

**What it unblocks.** Unblocks Phase 4 multi-entity/multi-region delivery without a future breaking migration: the relational foundation exists from Phase 1, so lighting up live sub-unit RLS isolation, multi-entity operations, or multi-region becomes a flag-flip-plus-feature build per named customer rather than a schema rewrite. Directly supports Q4 (the `legal_entity_id`-on-cases rule rides on this hierarchy).

**Confidence:** High.

**Still your call:** Explicit owner sign-off that "dormant foundation now, activate on demand" is the intended reading of locked decision #1's "depth delivered incrementally" — i.e. that shipping the full schema dormant and gating each depth layer behind a named customer is acceptable, versus a slower level-by-level live build.
